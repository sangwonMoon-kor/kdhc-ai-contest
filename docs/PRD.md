## 1. 제품 개요

### 1.1 한 줄 정의
순환근무 조직(한국지역난방공사)의 업무 기억을 사람이 아닌 **자리(직무)**에 귀속시키는
사내 업무 비서. 직원은 **Hello JARVIS?** 검색창에 묻고, 시스템은 사내 결재문서·규정·회의록·반려 이력을 근거로
답하며, 바로 초안 작성·사전점검·브리핑북으로 이어준다. 발령 첫날에는 브리핑북이 기다리고, 시즌이 오면 업무가 예보되며,
**평소에는 내 업무가 자동 분류되어 나만의 매뉴얼로 쌓인다.** ⭐

### 1.2 목표 (우선순위 순)
1. **사내 AI 경진대회 우승** — 실서비스 구현이 아니라 직원 공감과 발표 임팩트가 최우선
2. 심사위원·투표 직원이 데모 3분 안에 "저거 우리 부서에 있으면 쓰겠다"를 느끼게 할 것
3. 후속 개발 조직(IT부서·외부 파트너)이 문서만으로 착수 가능한 수준의 명세 제공 (기획서 4~7장)

### 1.3 핵심 가치
| 가치 | 설명 | 담당 기능 |
|---|---|---|
| 시간을 아낀다 | 인수인계 적응 단축, 반복 문서 작성 절감 | 기능 ①② |
| 직원을 지킨다 | 감사 지적·반려 사전 예방 — "나를 보호하는 AI" | 기능 ③ |
| 조직에 남긴다 | 문서 적재·태깅이 누적될수록 강해지는 구조 | 기반층 ⭐ |
| **나를 위한 도구** ⭐ | 물어보면 바로 답과 다음 액션을 얻고, 쓰다 보면 업무 기억이 쌓이는 구조 | Hello JARVIS + 기반층 |

## 2. 배경 및 문제 정의

### 2.1 문제
| 상황 | 현상 | 결과 |
|---|---|---|
| 발령 직후 | 인수인계가 형식적 문서 한 장으로 종료, 과거 문서 소재 파악 불가 | 전임자에게 전화 문의, 적응에 1~2개월 |
| 연간 반복업무 | 작년 문서를 못 찾거나 존재를 모름 | 70~80% 동일 문서를 매년 처음부터 작성 |
| 기안·결재 | 감사 지적·반려 이력이 개인 기억에만 존재 | 같은 지적·반려 반복, 심리적 부담 |
| 사내 AI 사용 제약 ⭐ | 외부 ChatGPT류 도구는 보안상 사용 곤란, 내부 문서도 알 수 없음 | 직원은 "회사 자료를 아는 AI"에 가장 목마름 |
| **평소** ⭐ | 업무 지식 정리는 "인수인계 때 몰아서 하는 일"로 인식 | 정리 유인 부재 → 데이터 공백의 근본 원인 |

### 2.2 전략적 포지셔닝 (경진대회 차별화 논리)
- **사무 AX 공백**: 공사 AX 로드맵(2025.8 발표, 2025.9 AX추진단 출범) 2단계 과제가 전부
  설비·현장 중심(안전·열수요예측·경제운전·설비진단) → 전 직원이 매일 하는 기안·인수인계·감사
  영역이 빈칸. 본 제안이 **사무 AX 첫 사례**이자 3단계(AI Agent)로 가는 디딤돌.
- **선행 사례 대비**: 포스코 P-GPT·한화 AIDA는 "검색(수동)", 한국도로공사 AI Draft는 "초안(단건 생성)"에
  머묾. 인천공항공사가 2025 공공 AI 왕중왕전 대상을 받은 공식은 **"예측 + 화면 표출"** → 기능 ②를
  알림에서 대시보드로 승격한 근거. SKT "1인 1 AI 에이전트"는 확산 비전(1자리 1메모리)의 근거.
- **순환근무·감사라는 공기업 고유 맥락**을 정면으로 다루는 사례는 국내 미확인.
- **도입 동기 방어** ⭐: 직원은 데이터를 쌓으려고 들어오지 않는다. "Hello JARVIS?"에 질문하고, 답변에서 초안·점검·브리핑을
  바로 열어 편해지기 때문에 쓴다. 그 사용 결과가 자동 분류되어 본인 매뉴얼(= 미래의 브리핑북)이 된다.

### 2.3 페르소나
**김민준 과장** — 1월 2일 자로 ○○지사 요금정산팀 발령. 전임자는 이미 본사로 이동.
발표·데모의 전 서사가 이 한 사람의 하루(발령 첫날 → 6개월 뒤 평소 ⭐)를 따라간다. 기능 나열 금지.

## 3. 제품 구조 ⭐ (v2.0 "3대 기능" → "기반층 + 3대 기능")

```
        ┌────────────────────────────────────────┐
        │ P0. Hello JARVIS? — 사내 근거 검색 진입점 │ ⭐
        │ 질문 → 근거 답변 → 초안/점검/브리핑 액션    │
        └───────────────┬────────────────────────┘
        ┌───────────────┴───────────────┐
        │ 기반층: 개인별 업무 분류         │ ⭐
        │ 결재·메일·회의록 → 업무 단위      │
        │ 자동 분류 → 개인 매뉴얼 상시 축적  │
        └──────┬────────────┬────────────┘
        ┌──────┴──────┐ ┌───┴────────┐ ┌──────┴──────┐
        │ ① 브리핑북   │ │ ② 업무 예보 │ │ ③ 사전 점검 │
        │ 과거·승계    │ │ 현재·초안   │ │ 보호·근거    │
        └─────────────┘ └────────────┘ └─────────────┘
```

### 3.0 P0 — Hello JARVIS? 사내 근거 검색 진입점 ⭐
- **정의**: 직원이 업무명을 몰라도 자연어로 물으면, 사내 결재문서·규정·회의록·반려 이력을 근거로 답하고
  근거 문서·검증상태·신뢰도와 함께 다음 업무 액션(초안 작성, 사전점검, 브리핑북, 업무 예보)을 제시하는 첫 화면.
- **UX 원칙**: 구글 첫 화면처럼 검색창 하나로 진입한다. 답변 화면은 챗봇보다 "근거 있는 업무 검색 결과"에 가깝게 구성한다.
  기술어(온톨로지, 파싱)는 숨기고, 근거·검증상태·다음 액션만 보인다.
- **채택 논리**: 외부 AI 사용이 어려운 공기업에서 직원이 가장 쉽게 공감하는 니즈는 "우리 회사 자료를 아는 AI에게 묻고 싶다"이다.
  질문은 업무 추가가 아니며, 답변 뒤 액션으로 초안·점검·브리핑 사용을 끌어낸다.
- **구현 목업**: `demo/jarvis.html` (`awesome-design-md`의 xAI/SpaceX/VoltAgent 문법을 차용한 JARVIS 디자인 시스템 기반).

### 3.1 기반층 — 개인별 업무 분류 · 밀착형 업무 매뉴얼 ⭐
- **정의**: AI가 각자의 결재·메일·회의록을 업무 단위로 자동 분류해 "밀착형 업무 매뉴얼"로
  상시 축적하는 층. 직원 입장에서는 매일 쓰는 개인 비서(내 업무 지도·문서 찾기·업무별 요령),
  조직 입장에서는 3대 기능의 데이터 기반.
- **핵심 논리**: 브리핑북 = 전임자의 밀착형 매뉴얼. 인수인계 준비가 별도 작업이 아니라
  평소 사용의 부산물이 된다.
- 기술적으로는 기획서 5장 "② 지식 계층(직무-문서 매핑 메타데이터)"의 사용자向 표출 개념 —
  아키텍처 신규 요소가 아니라 기존 계층의 재포지셔닝.

### 3.2 기능 ① 브리핑북 (과거 · 승계) — P1
- 발령 이벤트 시 직무 최근 2~3년 문서로 신임자용 요약 리포트 자동 생성.
- 구성: 직무 개요·핵심 업무 / 연간 업무 달력 / 최근 주요 이슈 3~5건 / 협조처 / 원문 링크.
- 품질: 모든 항목에 근거 문서 링크 필수, 근거 없으면 "문서 근거 없음" 명시 (환각 방지).
- 생성: 섹션별 분할 요약 후 통합(맵-리듀스), Gemma 4 중형(256K 컨텍스트). 인사연동은 P2(파일럿은 수동).

### 3.3 기능 ② 업무 일기예보 + 초안 준비 (현재 · 예보) — P1
- 과거 기안 시점 기반 반복 업무 도래를 달력·예보 카드로 선제 표출, 승인 시 작년 문서 기반 초안 즉시 생성.
- 예보 로직: **연 1회 이상 · 2년 연속** 반복 패턴 탐지 — LLM 불필요, 규칙 기반 배치.
  (∴ 데이터 요건: 전자결재 PDF 최소 2년치, 권장 2~3년 — 1년치로는 성립 불가)
- 대시보드: 월간 달력 + 주간 예보 카드(업무명·작년 기안일·근거 문서 수·D-day) + 부서 뷰/내 자리 뷰.
- 초안: 전년 문서 구조 분해 → 섹션 단위 생성(장문 일괄 생성 금지). 변경 필요 항목(연도·수치·일정)
  하이라이트. 수치·금액은 원문 인용만 허용(생성 금지). "AI 생성 초안 — 검토 필수" 고정 표기.

### 3.4 기능 ③ 사전 점검 (보호 · 점검) — P2
- 기안 제출 전 감사 지적사례·반려 이력·사규와 RAG 대조, 리스크 문구 하이라이트(주의/확인요망) + 근거 제시.
- 오탐 감수·미탐 최소화의 보수적 설정. **"문제 없음" 확정 표현 금지** — 최종 판단은 사람.
- 감사 데이터 민감성으로 P2 후순위(익명화·유형화 사례집으로 감사부서 협약 후 적재).
  경진대회 시연은 각색 샘플로 수행. 발표에서는 이 기능이 감정적 클라이맥스.

### 3.5 명시적 비범위 (Out of Scope)
- 자율 에이전트(도구 선택·자동 기안·자동 결재) — AX 3단계 시점 재검토
- 외부 클라우드 LLM API — 사내망 원칙 위배
- 암묵지(대인관계·구두 관행) 승계 — 문서에 없는 지식은 범위 밖임을 UI에 명시

## 4. 경진대회 데모 요구사항 (1차 산출물 — 구현 완료)

### 4.1 원칙
- 실서비스가 아닌 **시연용 프로토타입**. 전 데이터 하드코딩 샘플(실제 부서명·사업명·인명·감사사례
  미사용, 각색), LLM 호출 없이 사전 연출. 발표 시 정직 포지션: "화면은 프로토타입, 파이프라인은
  사내 인프라(5장 아키텍처)로 구현 가능."
- 단일 HTML(`demo/index.html`, `demo/jarvis.html`, `demo/admin.html`) + `demo/assets/` — 서버·네트워크 불필요, 더블클릭 구동.
- 해상도 1920×1080(16:9) 고정, 본문 최소 18px(뒷좌석 가독).

### 4.2 화면 구성 — 검색 목업 + 관리자 콘솔 + 7화면 ⭐

| `admin.html` ⭐ | 관리자 콘솔 | 직원 화면에서 숨긴 인제스트·파싱·온톨로지 관계·검증 상태·구조화 이벤트를 심사위원에게 공개. 팀원 엔진의 기능적 강점을 UI로 증명 | ✅ |

### 4.3 S0 상세 (영상 오프닝 + JARVIS 진입 — 기획서 미반영 ⭐)
`demo/assets/opening.mp4`가 첫 출근 장면을 짧게 보여준 뒤, 타이틀 "Hello JARVIS? 업무는 먼저 준비되게."로 전환된다.
첫 CTA는 `demo/jarvis.html`로 연결되어 사용자가 구글 첫 화면처럼 업무 질문을 입력하고, 답변·근거 문서·검증 상태·다음 액션을 확인한다.
이후 결과 액션에서 S1 브리핑북, S2 업무 일기예보, S3 초안, S4 사전점검으로 이어져 기존 본편 데모와 자연스럽게 결합된다.
상세 구현·검증 결과는 README와 `demo/tests/verify-s0-cutscene.js` 참조.

### 4.4 관리자 콘솔 상세 (신규 ⭐)
`demo/admin.html`은 직원용 제품 화면이 아니라 심사위원·관리자용 증명 화면이다. 직원에게는 "질문하면 근거와 다음 액션이 나온다"만 보이지만,
관리자는 같은 행동이 뒤에서 어떻게 인제스트, 파싱, 업무 분류, 관계 추출, 검증 상태, 민감정보 가드로 축적되는지 본다.
핵심 메시지는 **직원은 편해서 쓰고, 조직은 모르게 AX-ready 데이터가 쌓인다**이다.

콘솔 구성:
- 문서 16건·관계 49개·질의 4건·확인 필요 3건 메트릭.
- 수집 → 파싱 → 분류 → 관계 추출 → 검증 5단계 파이프라인.
- 업무·기안문·협조처·반복시점·반려사유 온톨로지 그래프.
- 직원 질문/초안/사전점검 행동이 구조화 이벤트로 남는 trace.
- 후보 → 작성자 확인 → 동료 확인 → 오너 승인 검증 단계.
- 사람 평가성 표현 차단 및 업무 표현 치환 가드.

### 4.5 S5 상세 (신규 ⭐)
진입 시 자동 재생 (`runManual()`): 어두운 시점 자막(2.2초, "발령 6개월 뒤 · 2026년 7월") →
피드 4건이 2.3초 간격 유입(전자결재·회신 메일·전자결재·회의록), 각 건이 서류 칩으로 해당 분류에
꽂히며 분류 카드 하이라이트 + 버전 bump(v2.3→v2.4) + 문서 수 +1 + 틱 사운드 → 클로징 오버레이
(~13초) → CTA(비전 보기 / 다시 보기). 매뉴얼 분류 4종(요금 정산 / 정산 민원 대응 / 정산시스템
전환 / 계획·보고)은 S1 핵심 업무 4개와 대응. 문구 데이터: `index.html`의 `MFEED` / `MBASE`.

### 4.6 조작 체계
`jarvis.html`: 예시 질문 클릭/검색 → 결과 화면, `Esc`/뒤로 버튼으로 홈 복귀, 결과 액션은 `index.html#s1~#s4`로 연결.

`admin.html`: `jarvis.html` 결과 화면 또는 직접 URL로 진입. 발표 클라이맥스에서 직원용 UX 뒤의 인제스트·온톨로지·검증 상태를 설명하는 화면.

`index.html`: `←`/`→` 화면 이동 · 숫자키 `0`~`6` 직행 ⭐ · `A` = S2 알림 재연출 · 딥링크 `#s0`~`#s6`
(+`#s4?auto=1` 점검 자동 실행) · S4 하단 버튼 "6개월 뒤 →"(S5 진입) ⭐

## 5. 실서비스 아키텍처 (요약 — 상세는 기획서 5장)

- **설계 원칙**: ① 에이전트가 아닌 고정 워크플로우(트리거→검색→생성→전달; 예측 가능성·구축 난도
  통제, AX 3단계에 에이전트로 승격 경로 확보) ② 기존 인프라 재사용(사내 Gemma 4 + RAG 플랫폼,
  신규 도입·외부 API 없음) ③ 근거 우선(원문 링크 강제, 근거 없는 문장 미출력)
- **5계층**: 데이터 수집·전처리(HWP/PDF 파서, 메타데이터 태거) → 지식 계층(문서 저장소 + 벡터
  인덱스 + 직무-문서 매핑 DB) → 워크플로우 계층(스케줄러/이벤트 핸들러/파이프라인) → 생성 계층
  (Gemma 4 중형 26B MoE 권장, 프롬프트 템플릿 3종) → 사용자 접점(웹 대시보드 + 알림 채널)
- 업무 일기예보는 LLM 미사용(규칙 기반 배치), 브리핑북·초안은 섹션 분할 생성 필수.
