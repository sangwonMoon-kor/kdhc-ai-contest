# 검증 계획 — 실제 반복 업무 리플레이

> 작성일: 2026-07-10
> 상태: **검증 방식 확정, 업무 선정 대기**
> 관련: [아이디어-만능입력.md](아이디어-만능입력.md) · [업무 건 중심 설계 스펙](superpowers/specs/2026-07-10-work-item-centered-product-design.md)

## 1. 왜 이 검증인가

업무 건 중심 설계(2026-07-10 확정)는 아직 상상으로 만든 구조다. 기능을 상상해서 넣는 대신,
**실제 반복 업무 한 건을 처음부터 끝까지 복원**해서 "업무를 끝내는 데 실제로 필요했던 정보"만
화면에 남긴다.

| 지금 고민하는 것 | 실제 업무에서 확인할 것 |
|---|---|
| 홈 입력에 무엇을 적는가 | 처음 받은 지시 원문 |
| Todo는 어떻게 생기는가 | 일을 진행하며 실제로 만든 할 일 |
| 어떤 기존 자료가 필요한가 | 실제로 열어본 작년 문서와 서식 |
| 메모가 Todo로 가는가 | 행동 메모인지, 사실·결정 메모인지 |
| 기안은 언제 시작하는가 | 어떤 정보가 모여야 초안을 쓸 수 있는지 |
| 업무 건에 무엇이 필요한가 | 완료까지 계속 확인한 정보 |

## 2. 절차 (5단계)

1. **업무 선정** — 최근 실제 수행한 반복 업무 1건. 조건: 작년 자료 있음 · 협조 부서 있음 ·
   기안/결과물 있음 · 전체 과정을 재현할 수 있는 크기
2. **재현** — 기존 방식 그대로 (메일·메신저·문서함·메모장·Todo 등 실제 도구)
3. **행동마다 이유 한 줄 기록** — 예: "기한이 불명확해서 팀장에게 재확인", "작년 문서 제목을
   몰라 세 번 검색", "운영부 답변을 메모했지만 기안에 옮기는 것을 잊음"
4. **완료 후 하나의 업무 건으로 재구성** — 지시 / 완료 조건 / Todo / 사용한 자료 / 협의 내용 /
   메모와 결정 / 최종 산출물 / 다음번 주의 사항
5. **그 정보만으로 업무 건 화면 설계** — 상상 기능 금지, 실제로 쓴 정보만

**과적합 방지**: 첫 사례로 구조를 만든 뒤, **다른 반복 업무 1~2건으로 공통 구조인지 확인**한다.
첫 사례만 보면 특정 업무 전용 화면이 된다.

## 3. 진행 방식

- 복원은 에이전트가 질문을 하나씩 던지며 인터뷰 형식으로 진행 (기억 복원 + 이유 기록)
- ⚠️ **보안**: 복원 과정의 실명·부서·금액·문서번호는 대화/로컬에만. **리포에 올릴 때는 각색**
  (기존 원칙: 데모 문구는 전부 각색 샘플)
- 타임박스: 복원 인터뷰는 1~2세션 내 완료 목표
- 첫 검증 업무는 데모의 펌프 정비공사를 억지로 쓸 필요 없음 — 본인이 실제 수행한 업무가 우선.
  화면 설계 후 데모 각색 시 유즈케이스(발전소 유지보수)로 번역

## 4. 이 라운드에서 함께 정리된 제안 (미확정 — 화면 설계 때 결정)

- **착지 방식**: 지시 입력 시 확인 카드를 거치지 않고 업무 건 화면에 직행, 요약("새 업무로
  정리했어요 · 할 일 5 / 자료 3 / 주의 2")은 상단 스트립으로. 중간 확인 카드는 화면 이동이
  없는 메모 저장에만
- **메모 삼분류**: 사실 메모 / Todo 후보 / 결정 메모. 행동 문장 발견 시 자동 이동 대신
  "[할 일에 추가]" 제안형. 메모 원문은 항상 보존
- **오분류 복구**: "질문이었나요? → 검색으로" 1탭 복구 유지
- **홈 입력창**: 모양은 검색창 유지, 플레이스홀더를 지시 예시("팀장님이 다음 주까지 …올리래")로
  교체, 예시 칩은 지시·질문·메모 혼합
- **입력별 강조**: 같은 업무 건 화면에서 입력 종류에 따라 처음 강조되는 영역만 변경
  (지시→개요·Todo, 질문→자료·답, 메모→진행 기록, 초안 요청→기안 영역)
