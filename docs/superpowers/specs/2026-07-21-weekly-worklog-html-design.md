# 7월 4주차 작업 기록 HTML 설계

## 1. 목적

현재까지 진행한 직무 메모리 프로젝트의 작업 내용, 주요 결정, 검증 결과, 이어서 작업하는 방법을 한 화면에서 확인할 수 있는 주간 작업 기록을 만든다. 문서는 개발자가 아닌 팀원도 이해할 수 있어야 하며, 로컬에서 바로 열 수 있고 GitHub에서도 파일 구조를 확인할 수 있어야 한다.

## 2. 산출물

- 최종 파일: `docs/7월 4주차.html`
- 실행 방식: 파일을 브라우저로 직접 열거나 저장소의 로컬 서버를 통해 연다.
- 의존성: 외부 폰트, CDN, JavaScript 패키지를 사용하지 않는 단일 HTML 파일로 만든다.
- 저장 위치: 기존 `docs` 폴더를 사용하며 별도의 새 문서 폴더는 만들지 않는다.

## 3. 독자와 사용 상황

주요 독자는 프로젝트 담당자, 경진대회 팀원, 다른 컴퓨터에서 작업을 이어받는 사람이다. 문서를 열었을 때 다음 질문에 바로 답할 수 있어야 한다.

1. 직무 메모리는 어떤 문제를 해결하려는 프로젝트인가?
2. 지금까지 어떤 화면과 기능을 구현했는가?
3. OpenAI API는 어떻게 연결했고 현재 어떤 상태인가?
4. 실제 회사 데이터는 어디까지 검토했고 어떤 보안 결정을 내렸는가?
5. 다음 작업자는 어떤 순서로 이어서 작업하면 되는가?

## 4. 정보 구조

HTML은 다음 순서로 구성한다.

1. **표지 요약**
   - 문서명, 기록 기간, 프로젝트명
   - 현재 상태를 `완료`, `검토`, `보류` 배지로 요약
2. **프로젝트 목표**
   - 회사 자료를 업무 건 중심으로 연결해 검색, 할 일, 근거, 기안으로 이어주는 목적
   - 현재 결과물이 경진대회용 로컬 프로토타입이라는 경계
3. **이번 기록의 핵심 성과**
   - 홈과 업무 작업대
   - 업무 중심 데이터 구조
   - OpenAI 검색 연결
   - `.env` 기반 로컬 키 관리
4. **작업 타임라인**
   - 화면과 오프닝 개선
   - 업무 작업대 목업
   - AI 검색 서버 및 클라이언트 연결
   - 실제 API 검증과 모델 변경
   - 실제 유지보수 데이터 검토
5. **현재 제품 흐름**
   - 홈 질문 → AI 판단 → 업무 선택 → 근거 표시 → 할 일/메모/기안 연결
   - AI 실패 시 샘플 응답으로 폴백하는 동작
6. **OpenAI API 운영 방법**
   - `.env` 항목 이름만 표시
   - `npm run demo:ai` 실행과 종료 방법
   - API 키를 채팅이나 Git에 올리지 않는 원칙
7. **실제 유지보수 데이터 검토**
   - 입력 파일명과 파일 크기 수준
   - 7개 상위 절차 영역과 활용 후보
   - 변환 판정 `REVIEW`, 잔여위험 `MEDIUM`, 수동 검토 필요
   - 실제 데이터는 로컬 데모에서만 사용하기로 한 결정
   - 외부 OpenAI API 전송 범위는 아직 결정하지 않았음을 표시
8. **Git 상태와 주요 커밋**
   - 작업 브랜치와 원격 브랜치
   - AI 검색 관련 핵심 커밋
   - `.env`는 추적되지 않는다는 설명
9. **이어가기 체크리스트**
   - 저장소 이동, 의존성 설치, `.env` 준비, 서버 실행, 화면 확인
   - 실제 데이터 통합 전 확인해야 할 보안 결정

## 5. 시각 설계

- 흰색과 연한 회색 바탕에 파란색을 주요 강조색으로 사용해 현재 제품과 톤을 맞춘다.
- 상단에는 큰 제목과 상태 요약 카드를 배치한다.
- 본문은 최대 폭을 제한해 긴 문장을 읽기 쉽게 한다.
- 성과는 카드, 과정은 세로 타임라인, 결정 사항은 배지와 강조 상자로 표현한다.
- 넓은 화면에서는 2열 카드, 작은 화면에서는 1열로 자동 전환한다.
- 인쇄 시 배경과 정보 계층이 유지되도록 인쇄용 스타일을 포함한다.
- 장식보다 정보 전달을 우선하며 애니메이션은 사용하지 않는다.

## 6. 보안과 데이터 경계

HTML에 다음 내용을 포함하지 않는다.

- OpenAI API 키 또는 키 일부
- `.env`의 실제 값
- 유지보수 절차서 원문과 상세 표
- 조직, 사업장, 설비 또는 업체를 재식별할 수 있는 값
- 로컬 파일의 원문 전체 경로
- 비식별 대응표와 미검증 도해

실제 데이터 검토 내용은 판정, 문서 구조, 활용 가능 영역, 결정 사항만 요약한다. `sanitized.md`와 `변환결과_REVIEW.json`은 저장소로 복사하지 않는다.

## 7. 링크 정책

- 저장소 안의 문서와 데모에는 상대 경로를 사용한다.
- 실행 명령은 복사 가능한 코드 블록으로 제공한다.
- GitHub 브랜치 주소는 일반 링크로 제공하되 인증 정보는 포함하지 않는다.
- 존재하지 않는 문서나 예정 파일로 연결하지 않는다.

## 8. 검증 기준

완료된 HTML은 다음 조건을 만족해야 한다.

1. 브라우저에서 로컬 파일로 열었을 때 오류 없이 표시된다.
2. 한글이 UTF-8로 정상 표시된다.
3. 데스크톱과 모바일 폭에서 가로 스크롤이 발생하지 않는다.
4. 제목, 표지, 타임라인, 보안 결정, 이어가기 체크리스트가 모두 존재한다.
5. API 키 형태의 문자열과 실제 유지보수 본문이 포함되지 않는다.
6. HTML 유효성의 기본 오류와 깨진 내부 링크가 없다.
7. Git diff 검사와 프로젝트의 기존 OpenAI 검증을 통과한다.

## 9. 범위 제외

- 실제 유지보수 데이터의 제품 통합
- 실제 데이터의 OpenAI API 전송
- GitHub Pages 배포
- 기존 제품 화면의 디자인 변경
- 주간 기록 자동 생성 기능

이번 작업은 현재 상태를 안전하고 읽기 좋게 전달하는 정적 작업 기록 문서 제작까지만 포함한다.
