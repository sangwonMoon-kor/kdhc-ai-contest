# ON_메모리 제품 구조와 업무 수행 메모리 설계

> 상태: 사용자 승인 완료
>
> 작성일: 2026-07-23
>
> 목적: 확정된 홈 UI를 기준으로 공통 사이드바, 일정, 내 업무, 클라우드, 업무 작업대와 완료 후 재사용 흐름을 하나의 제품으로 정의한다.

## 1. 설계 결론

ON_메모리는 기능별 AI 도구 모음이나 문서 검색기가 아니다. 직원이 맡은 일을 놓치지 않고 끝낼 수 있도록 준비하고, 그 과정에서 나온 판단과 결과를 다음 담당자에게 남기는 **직무 중심 AI 업무 비서**다.

제품의 중심 단위는 사용자도 문서도 아닌 **업무 건**이다. 대표 업무 건은 `2026 정기점검보수 기본계획 수립`이며, 업무 건 하나가 다음 정보를 연결한다.

```text
업무 건
├─ 언제 해야 하는가
├─ 무엇을 참고해야 하는가
├─ 어떤 기준을 지켜야 하는가
├─ 무엇을 만들어야 하는가
├─ 진행하면서 남긴 메모
├─ 업무 중 확정된 결정
├─ 변경된 내용과 이유
└─ 최종 결과물
```

직원은 평소처럼 자료를 올리고, 일정을 확인하고, 자연어로 생각을 남기며 결과물을 만든다. AI는 그 활동을 현재 업무에 연결하고 구조화한다. 업무가 완료되면 과정과 결과가 해당 직무에 남아 다음 연도 또는 다음 담당자가 재사용한다.

## 2. 공통 제품 셸

### 공통 사이드바

좌측 사이드바는 홈뿐 아니라 모든 화면에서 유지한다. 메뉴 이름은 아이콘과 함께 항상 표시한다.

```text
홈
내 업무
일정
클라우드
```

- `홈`: 자연어 입력과 가까운 2주 업무 브리핑
- `내 업무`: 개인 일정과 자신이 관련된 작업을 모은 개인 업무 피드
- `일정`: 내 업무, 우리 과, 부서 전체 작업을 조합해 보는 작업 달력
- `클라우드`: 현재 담당 직무에 귀속되는 개인형 업무 라이브러리

초안 작성, 사전 점검, 문서 검색, 지식 지도는 공통 메뉴로 노출하지 않는다. 선택한 업무에 필요한 순간 업무 작업대 안에서 제공한다.

### 시각 기준

현재 확정된 홈 화면을 모든 화면의 시각 정본으로 사용한다.

- 거의 흰색으로 인식되는 밝은 배경
- 가장자리의 옅은 빨강·주황 조명
- 흰색 글래스 표면과 얇은 경계선
- 둥근 공통 사이드바와 넉넉한 여백
- 선명한 빨강은 현재 위치와 핵심 행동에만 사용
- 초록은 확인 완료와 개인 일정처럼 확정된 긍정 상태에 사용
- 회색은 아직 열리지 않은 단계와 부서 전체 보조 정보를 표현

## 3. 홈

홈은 채팅 화면이나 전체 현황판이 아니라 **내게 가까운 업무를 준비하는 입구**다.

### 자연어 입력

- 안내 문구는 `지금 어떤 생각을 하시나요?`를 유지한다.
- 직원은 지시, 질문, 메모, 일정을 구분하지 않고 입력한다.
- AI는 대상 업무와 의도를 판단해 업무 생성, 작업 기록, 개인 일정, 질문 답변으로 처리한다.
- 결과는 `정기점검보수 기본계획의 일정으로 기록했습니다 · 되돌리기`처럼 대상과 실제 변경 내용을 알려준다.
- 대상 업무나 날짜 의미가 불명확할 때만 한 번 확인한다.

### 앞으로 2주

- 홈 달력은 `내 담당 작업 + 내 개인 일정`만 표시한다.
- 우리 과와 부서 전체 필터는 홈에 두지 않는다.
- 작업은 시작일과 종료일을 잇는 기간 막대로 표시한다.
- 주요 마감과 단계는 기간 막대 안이나 해당 날짜에 표시한다.
- 자연어 메모에 날짜가 있으면 개인 일정 후보로 연결한다.
- 개인 일정은 달력에 직접 추가하거나 수정할 수 있다.
- 일정 항목을 선택하면 작업은 업무 작업대로, 개인 일정은 일정 상세로 이동한다.

## 4. 일정

한난에는 별도 사내 캘린더가 없으므로 일정 화면은 보조 기능이 아니라 핵심 제품 영역이다. 일반 소비자 캘린더보다 **부서 작업 일정**을 중심으로 설계한다.

### 조직과 기본 범위

조직 계층은 다음과 같이 본다.

```text
부서 → 과 → 담당자
```

일정 화면의 기본값은 `내 업무`만 켜진 상태다. 다음 세 범위는 각각 독립적으로 켜고 끌 수 있다.

- `내 업무`
- `우리 과`
- `부서 전체`

두 개 또는 세 개를 동시에 선택할 수 있으며 결과는 합집합으로 표시한다. 같은 작업이 여러 범위에 포함되더라도 한 번만 표시한다. 가까운 담당 범위의 시각 스타일을 우선한다.

범위는 서로 다른 데이터 복사본이 아니라 동일 작업 집합을 조회하는 독립 레이어다. 예를 들어 내 작업이 `우리 과`와 `부서 전체`에도 포함되더라도 중복 카드를 만들지 않고 `내 업무 → 우리 과 → 부서 전체` 순서로 가까운 관계의 표현을 적용한다.

- 내 담당 작업: 선명한 빨강
- 우리 과 작업: 옅은 빨강
- 부서 전체 작업: 중성 회색
- 개인 일정: 초록

`부서 전체`만 켜면 부서 전체 작업을 볼 수 있다. 여러 범위를 함께 켜면 내 작업을 가장 강하게 표시하고, 우리 과와 나머지 부서 작업을 보조 계층으로 구분한다.

### 작업 표현과 조작

- 작업 전체는 시작일과 종료일을 잇는 기간 막대로 표현한다.
- 준비, 현장 작업, 제출, 완료처럼 중요한 단계와 마감만 기간 위에 표시한다.
- 월간 보기를 기본으로 하고 필요하면 주간 보기로 전환한다.
- 달력에서 작업을 선택하면 해당 업무 작업대를 연다.
- 개인 일정은 빈 날짜를 선택해 직접 등록할 수 있다.
- 작업 기간이나 개인 일정 변경은 되돌릴 수 있어야 한다.

달력은 OpenProject의 작업 기간 표현과 담당자 필터, Todoist의 단순한 일정 조작 방식을 참고하되 ON_메모리의 업무 건과 직무기록 모델에 맞게 단순화한다.

## 5. 클라우드

클라우드는 공유 드라이브나 완료 업무 현황판이 아니다. 직원이 자신의 업무를 위해 가진 과거 자료와 현재 자료를 편하게 넣어두는 **직무 귀속 개인형 라이브러리**다.

### 화면과 실제 귀속

- 화면 명칭은 `내 클라우드`로 표시한다.
- 사용 중에는 현재 담당자의 개인 업무 공간처럼 보인다.
- 실제 데이터는 개인 계정이 아니라 현재 맡고 있는 직무에 귀속된다.
- 담당자가 이동하면 기존 클라우드는 이전 직무에 남는다.
- 새 담당자는 해당 직무의 연도·업무별 라이브러리에서 시작한다.
- 이동한 직원은 새 직무에 귀속된 클라우드를 사용한다.

### 폴더 구조

C드라이브처럼 익숙한 폴더 트리와 파일 목록을 제공한다. 기본 구조는 연도와 업무를 중심으로 한다.

```text
내 클라우드
├─ 2026
│  ├─ 정기점검보수
│  │  ├─ 기준·참고 자료
│  │  ├─ 진행 자료
│  │  └─ 최종 결과물
│  ├─ 순환수 펌프 정비
│  └─ 차단기 예비품 구매
├─ 2025
├─ 2024
├─ 공통 서식·기준
└─ 미분류
```

사용자가 만든 폴더 구조를 유지하며 AI가 강제로 재작성하지 않는다. AI가 새 폴더를 제안하거나 자료를 이동한 경우 사용자가 되돌릴 수 있다.

### 업로드와 AI 정리

```text
편하게 업로드
→ AI가 폴더·태그·연결 작업 제안
→ 사용자 확인
→ 연도·업무별 자동 정리
```

- 사용자는 파일이나 폴더를 위치 고민 없이 올릴 수 있다.
- AI는 원문 내용, 연도, 관련 업무, 자료 상태를 분석한다.
- `2026 / 정기점검보수 / 진행 자료`, `#기본계획`, `작업 #W-2026-008`처럼 정리안을 제안한다.
- 사용자가 확인하면 같은 기준으로 라이브러리를 정리한다.
- 처음 보는 정리 패턴이나 확신이 낮은 연결은 다시 확인하고, 이미 확인된 동일 패턴은 자동 적용한 뒤 되돌리기를 제공한다.
- 원본 파일은 보존하고 자동 분류와 이동은 되돌릴 수 있다.
- 읽지 못한 파일도 버리지 않고 `내용 확인 필요` 상태로 보관한다.

직무 이동 시 별도의 인계 후보 생성이나 마지막 파일 선별 단계는 두지 않는다. 평소 확인된 정리가 그대로 다음 담당자의 시작 데이터가 된다.

## 6. 내 업무

내 업무는 작업 카드 모음이 아니라 **개인 업무 피드**다. 다음 항목을 한곳에서 시간 순서와 중요도에 따라 보여준다.

- 홈이나 달력에서 직접 저장한 개인 일정
- 내가 담당한 작업
- 참여자로 연결된 작업
- 참고자로 연결된 작업

### 필터와 정보

- `전체`
- `확인 필요`
- `관련 작업`
- `내 일정`
- `완료`

각 작업 항목에는 날짜, 관계 유형, 현재 단계, 다음 행동, 준비된 자료, 확인 필요 여부를 표시한다. 개인 일정은 수정과 완료를 제공한다.

### 즉시 행동

작업 목록에서 화면을 이동하지 않고 다음 행동을 실행할 수 있다.

- `검토`: AI가 준비한 일정, 기준, 초안 또는 확인 요청을 검토한다.
- `코멘트`: 작업에 자연어 직무기록을 남긴다.
- `열기`: 전체 업무 작업대로 이동한다.

코멘트는 관계자 대화방에 보내는 채팅이 아니라 해당 작업에 귀속되는 업무 수행 데이터다.

## 7. 업무 작업대

업무 작업대는 ON_메모리의 중심 화면이다. Todo, 문서, 초안 기능을 나열하는 대신 **최종 결과물에 도달하기까지의 사고 흐름**을 보여준다.

### 업무 사고 흐름

```text
언제 해야 하는가
→ 무엇을 참고해야 하는가
→ 어떤 기준을 지켜야 하는가
→ 무엇을 만들어야 하는가
→ 최종 결과물
```

진행하면서 남긴 메모, 업무 중 확정된 결정, 변경된 내용과 이유는 별도의 마지막 단계가 아니다. 모든 사고 단계에서 계속 발생하고 현재 단계와 작업에 연결된다.

### 단계 표현

질문형 문구를 줄이지 않는다. 문장 안 핵심어를 굵게 처리한다.

- **언제** 해야 하는가
- **무엇을 참고**해야 하는가
- **어떤 기준**을 지켜야 하는가
- **무엇을 만들어야** 하는가
- **최종 결과물**

현재 단계만 빨강으로 강조하고, 확인 완료 단계는 초록, 예정 단계는 중성 회색으로 표시한다. 짧은 추상 명칭보다 직원이 바로 이해할 수 있는 질문형 표현을 우선한다.

### 단계 진행 규칙

업무 사고 흐름을 수동 마법사로 만들지 않는다.

1. AI가 일정, 클라우드, 과거 업무에서 각 단계 내용을 미리 채운다.
2. 출처가 없거나 확실하지 않은 항목만 현재 확인 대상으로 강조한다.
3. 사용자는 일정, 적용 기준, 결과물 구조처럼 중요한 판단을 확인한다.
4. 확인되면 다음 단계가 자동으로 활성화된다.
5. 이전 단계는 언제든 다시 열어 수정할 수 있다.
6. 네 사고 단계가 충족돼야 최종 결과물을 확정할 수 있다.

여기서 `충족`은 반드시 자료가 존재한다는 뜻이 아니다. `관련 자료 없음`, `적용 기준 없음`, `담당자 판단으로 해당 없음`도 근거 또는 사유와 함께 확인되면 완료된 단계로 본다. 데이터 부재 때문에 실제 업무 완료를 막지 않는다.

### 화면 구성

- 상단: 업무명, 작업 기간, 담당 관계, 완료 상태
- 사고 흐름: 다섯 단계와 현재 위치
- 현재 단계: 확인할 내용, AI 제안, 원문 근거, 확인 행동
- 연결된 맥락: 과거 문서, 최신 기준, 관련 코멘트
- 수행 기록: 메모, 결정, 변경과 이유를 시간 순서로 표시
- 자연어 입력: `이 업무에서 판단한 내용을 기록`
- 결과물: 초안, 사전 점검, 최종 결과물

각 Todo나 기준은 필요한 근거와 떨어뜨리지 않는다. AI가 만든 초안은 완성된 결재문이 아니라 검토 가능한 출발점이며, 적용 기준과 과거 결과물의 출처를 함께 보여준다.

## 8. 자연어 코멘트와 직무기록

직원에게 기록 유형, 태그, 날짜 필드를 선택하게 하지 않는다.

```text
직원이 자연어로 기록
→ AI가 현재 작업 맥락에서 구조화
→ 대상 작업과 반영 결과 안내
→ 되돌리기 제공
```

AI가 분류하는 내용은 다음과 같다.

- 진행 메모
- 확정된 결정
- 변경된 내용과 이유
- 후속 일정
- 확인할 자료
- 다음 행동

AI가 충분히 확신하면 즉시 기록하고 `되돌리기`를 제공한다. 대상 작업이나 날짜 의미가 불명확할 때만 한 번 확인한다.

### 보존 정보

사용자에게 보이는 작성자 메타데이터는 다음 두 가지로 단순화한다.

- 작성자 이름
- 작성 시점

기록 자체는 현재 작업과 단계 안에서 생성되므로 시스템 관계로 다음 정보를 보존한다.

- 연결된 작업 ID
- 당시 작업 단계
- 연결 자료
- 작성자가 쓴 원문

별도의 직무명은 저장하거나 표시하지 않는다. 작업 연결 자체가 직무 맥락을 제공한다. AI는 원문을 대체하지 않고 검색과 재사용을 위한 구조화 정보를 덧붙인다.

다음 연도 같은 작업이 시작되면 관련 단계에서 과거 담당자 코멘트의 원문, 작성자, 시점, 연결 자료를 함께 제시한다. 이는 사람을 평가하는 데이터가 아니라 업무 판단의 근거다.

## 9. 업무 완료와 계획 대비 변화

회사 업무는 시간이 더 들거나 비용이 증가하더라도 결국 완료된다. 제품은 `성공 / 실패` 또는 `명예 / 불명예`로 사람과 업무를 평가하지 않는다.

### 완료 규칙

- AI는 완료 준비 상태와 빠진 항목을 알려준다.
- 최종 결과물은 작업에 연결되어야 한다.
- 담당자가 결과물과 수행 기록을 확인한다.
- 최종 완료는 AI가 아니라 담당자가 직접 확정한다.
- 완료 이후 과정과 결과를 하나의 직무기록으로 묶는다.

최종 결과물은 반드시 하나의 문서 파일일 필요는 없다. 기안 최종본, 결재 문서 참조, 현장 작업 결과 기록, 확정된 데이터 묶음처럼 해당 업무의 완료를 증명하는 결과를 허용한다.

### 완료 후 남기는 정보

완료 상태는 하나의 `완료`로 표시하고 계획과 달라진 사실을 별도 태그와 기록으로 남긴다.

- 실제 완료일과 계획 대비 일정 차이
- 추가 비용 발생 여부와 이유
- 작업 범위 변경과 이유
- 적용 기준 변경
- 실제로 사용한 자료
- 담당자 코멘트와 확정된 결정
- 최종 결과물

예시는 다음과 같다.

```text
완료
├─ 일정: 계획보다 12일 지연
├─ 비용: 추가 비용 발생
├─ 범위: 현장 상황으로 작업 범위 확대
├─ 변경 이유: 밸브 진동 수치가 기준 초과
└─ 최종 결과물: 정기점검보수 기본계획 최종본
```

완료를 차단하거나 실패로 바꾸지 않는다. 지연, 비용, 범위 변경을 숨기지 않고 다음 업무를 위한 데이터로 남긴다.

## 10. 핵심 데이터 모델

### 조직

```text
Department
└─ Section
   └─ Person
```

- `Department`: 부서
- `Section`: 과
- `Person`: 현재 사용자와 작성자
- `Role`: 클라우드와 직무기록이 귀속되는 직무 자리

### 업무와 일정

- `WorkItem`: 업무 ID, 제목, 기간, 상태, 담당·참여·참조 관계, 현재 사고 단계
- `ScheduleItem`: 작업 기간, 주요 마감, 개인 일정, 확정 여부
- `WorkStageState`: 질문형 단계, AI 제안, 근거, 사용자 확인 상태

### 자료와 기록

- `RoleLibrary`: 직무에 귀속되는 클라우드 루트
- `SourceAsset`: 파일 원본, 연도, 폴더, 태그, 연결 업무, 인덱싱 상태
- `WorkRecord`: 원문, 작성자, 작성 시점, 작업 단계, AI 분류, 연결 자료
- `FinalArtifact`: 최종 결과물과 버전
- `CompletionSnapshot`: 완료일, 계획 대비 변화, 실제 사용 자료, 최종 결과물

개인 일정은 개인에게 귀속되지만 작업 일정, 업무 기록, 클라우드 자료와 완료 스냅샷은 업무 또는 직무에 귀속된다.

## 11. 프론트엔드와 엔진 역할

### UI 정본

- `kdhc-ai-contest/product-ui`를 최신 UI·UX 정본으로 유지한다.
- 팀원 저장소의 `jikmu-memory/service/public`은 동기화된 실행 사본이다.
- 발표와 녹화에서는 두 저장소를 따로 보여주지 않고 하나의 ON_메모리 제품으로 제공한다.

### 현재 엔진 기능의 활용

- 반복 시점과 예보: 홈 2주 일정과 작업 일정 후보
- 문서·온톨로지 질의: 참고자료, 적용 기준, 과거 판단 근거
- 초안 생성: `무엇을 만들어야 하는가` 단계의 출발점
- 사전 점검: 적용 기준과 과거 문서를 이용한 결과물 검토
- 문서 인제스트와 추출: 클라우드 업로드 자료의 내용 파악
- 힌트 구조화: 자연어 코멘트의 메모·결정·변경·후속 일정 연결

### 새 UI 모델에서 필요한 어댑터

현재 엔진에 없는 다음 정보는 발표용 상태 모델 또는 새 API 어댑터로 제공한다.

- 부서·과·담당자 관계
- 일정 범위 독립 레이어
- 개인 일정
- 직무 귀속 클라우드 폴더
- 담당·참여·참조 작업 관계
- 사고 단계 확인 상태
- 완료 스냅샷과 계획 대비 변화

1차 구현은 대표 시연 한 건을 완전하게 연결하는 것을 우선하고, 전사 권한·영속화·실시간 협업을 과장하지 않는다.

## 12. 신뢰와 예외 처리

- 일정, 기준, 답변과 AI 제안은 원문 근거를 제공한다.
- AI가 근거를 찾지 못하면 `확인 필요` 또는 `관련 자료 없음`으로 표시한다.
- 날짜, 비용, 규정 내용을 임의로 생성하지 않는다.
- 자연어 대상 작업이나 날짜가 애매하면 한 번만 확인한다.
- AI가 적용한 일정, 분류, 폴더 이동은 되돌릴 수 있다.
- 읽지 못한 파일은 원본을 보존하고 재시도할 수 있게 한다.
- 같은 작업이 여러 일정 범위에 포함돼도 중복 표시하지 않는다.
- 작성자 코멘트의 원문을 AI 요약으로 덮어쓰지 않는다.
- 지연과 추가 비용은 사람 평가가 아니라 업무의 계획 대비 변화로 기록한다.
- 사내 평가와 감시에 이용될 개인 성과 점수나 순위를 만들지 않는다.

## 13. 대표 발표 시나리오

대표 업무는 `2026 정기점검보수 기본계획 수립` 한 건으로 고정한다.

1. 홈에서 `다음 주까지 정기점검보수 기본계획을 준비해야 해`라고 입력한다.
2. AI가 업무 건과 2주 일정 후보를 만들고 직원이 결과를 확인한다.
3. 일정 화면에서 내 업무를 기본으로 보고 필요하면 우리 과와 부서 전체 작업을 추가한다.
4. 클라우드에 과거 한글·엑셀·PDF 자료를 올리고 AI 정리안을 확인한다.
5. 내 업무에서 관련 작업과 개인 일정을 함께 보고 작업을 검토한다.
6. 업무 작업대에서 `언제`, `참고`, `기준`, `만들 것`의 사고 흐름을 확인한다.
7. 직원이 `성적서를 먼저 확인하고 업체 서류는 7월 30일까지 받기로 함`이라고 자연어로 기록한다.
8. AI가 원문을 보존하며 결정, 변경 이유, 일정, 확인 자료로 구조화한다.
9. 직원이 초안과 사전 점검을 거쳐 최종 결과물을 연결한다.
10. 담당자가 업무를 완료하고 일정 지연, 추가 비용, 범위 변경을 계획 대비 변화로 확인한다.
11. 다음 해 같은 업무에서 과거 결과물과 실제 담당자의 판단이 필요한 단계에 먼저 제시된다.

발표 메시지는 다음 한 문장으로 수렴한다.

> 일한 만큼 준비되고, 다음 사람은 빈 화면이 아니라 업무의 기억에서 시작합니다.

## 14. 검증 기준

### 제품 셸

- 모든 주요 화면에서 `홈 / 내 업무 / 일정 / 클라우드` 사이드바가 유지된다.
- 메뉴 이름이 항상 보이고 현재 위치만 빨강으로 강조된다.
- 기존 홈의 시각 언어가 나머지 화면에도 일관되게 적용된다.

### 홈과 일정

- 홈에는 내 담당 작업과 개인 일정만 표시된다.
- 일정의 세 범위를 독립적으로 선택할 수 있다.
- 여러 범위를 선택하면 합집합을 중복 없이 표시한다.
- 작업 기간, 주요 마감, 개인 일정의 시각 계층이 구분된다.

### 클라우드

- 파일을 위치 선택 없이 업로드할 수 있다.
- AI가 연도·업무 폴더, 태그, 연결 작업을 제안한다.
- 확인 후 정리가 적용되고 원본 보존과 되돌리기가 가능하다.
- 직무가 바뀌어도 기존 자료는 이전 직무 라이브러리에 남는다.

### 내 업무와 작업대

- 개인 일정과 담당·참여·참조 작업이 한 피드에 표시된다.
- 작업 목록에서 검토와 코멘트를 바로 실행할 수 있다.
- 작업대에 질문형 사고 단계가 원래 문구로 표시된다.
- AI가 찾은 자료와 기준에 원문 근거가 연결된다.
- 자연어 코멘트가 현재 작업과 단계에 연결된다.
- 작성자 원문, 이름, 시점이 다음 업무에서도 보존된다.

### 완료와 재사용

- 최종 업무 완료는 담당자가 직접 확정한다.
- 완료 상태와 계획 대비 변화가 분리되어 표시된다.
- 완료 기록에 실제 사용 자료, 결정, 변경 이유와 최종 결과물이 포함된다.
- 다음 연도 업무에서 관련 단계에 과거 담당자의 실제 판단이 제시된다.

## 15. 1차 구현에서 제외하는 범위

- AI 기능별 독립 메뉴
- 관계자 실시간 채팅과 알림 시스템
- 부서 관리자용 성과·감시 대시보드
- 사람 평가 점수와 순위
- 공유 클라우드 루트와 복잡한 권한 관리
- 직무 이동 시 별도의 인계 후보 선별 화면
- 전사 시스템 수준의 캘린더 충돌 해결
- 모든 업무 유형을 위한 범용 워크플로 편집기
- 실서비스 수준의 결재 상신과 외부 시스템 쓰기

1차 구현은 실제 업무 전체 시스템을 흉내 내는 것이 아니라, 하나의 대표 업무가 시작되어 사고 과정을 거치고 완료된 뒤 다음 담당자에게 재사용되는 흐름을 높은 완성도로 증명한다.
