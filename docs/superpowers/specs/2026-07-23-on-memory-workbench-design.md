# ON_메모리 업무 작업대 구체화 설계

## 1. 목적

업무 작업대는 선택한 업무 한 건의 일정, 기준, 과거 기억, 결과물, 진행 과정과 완료 기록을 한곳에서 관리하는 제품의 메인 공간이다.

사용자는 `내 업무`의 진행 목록, 홈 일정, 조직 일정 또는 자연어 입력에서 업무 한 건을 선택해 작업대에 들어온다. 작업대 안에서 다시 업무를 선택하지 않는다.

작업대가 답해야 하는 질문은 다음 여섯 가지다.

1. 언제 해야 하는가
2. 무엇을 참고해야 하는가
3. 어떤 기준을 지켜야 하는가
4. 무엇을 만들어야 하는가
5. 지금까지 어떻게 진행됐는가
6. 언제, 어떤 결과로 완료됐는가

## 2. 확정된 제품 원칙

- `내 업무`의 기본 목록은 현재 수행 중인 업무다.
- 업무 작업대는 업무 한 건에 종속된다.
- 헤드라인에는 `진행 중`과 같은 공통 상태를 표시하지 않는다.
- 업무 단계는 `설계`, `계약`, `시공`, `준공`, `완료`와 같은 세부 단계만 표시한다.
- 완료된 업무는 `내 업무 > 완료`에 남는다.
- 완료 업무의 문서와 기록 묶음은 클라우드에서도 조회한다.
- 공식 기준과 과거·개인 업무 메모리를 분리한다.
- AI가 제안한 일정, 분류, 초안은 사용자가 확인하기 전까지 확정 정보가 아니다.
- 사용자 원문과 출처는 보존한다.
- 자료나 날짜가 없으면 임의 값을 만들지 않는다.

## 3. 진입 구조

업무 작업대는 다음 경로에서 열린다.

- `내 업무 > 진행`에서 업무 카드 선택
- 홈의 2주 일정에서 업무 막대 선택
- 조직 일정에서 업무 막대 선택
- 자연어 입력이 기존 업무와 연결됐을 때 해당 업무로 이동

업무 ID는 모든 진입 경로, 일정, 문서, 기록, 결과물과 완료 묶음에서 동일하게 유지한다.

## 4. 화면 구조

작업대는 탭으로 역할을 분리하지 않고 한 화면을 위에서 아래로 읽는 단일 업무 문서형을 사용한다.

### 4.1 헤드라인

헤드라인은 다음 정보를 표시한다.

- 작업명
- 세부 단계
- 일정 기준의 이름과 날짜
- D-day
- 담당 조직 또는 담당자
- `업무 완료` 행동

#### D-day 기준

- 계약 전 업무는 사용자가 확정한 설계 발송일을 기준으로 계산한다.
- 계약 후 업무는 계약 정보에서 확인된 준공일을 기준으로 계산한다.
- 기준일이 없으면 `일정 미정`과 `날짜 추가` 행동을 표시한다.
- AI가 과거 자료에서 찾은 날짜는 `일정 후보`로만 제안한다.
- 사용자가 확정하거나 계약 데이터에서 확인된 날짜만 헤드라인 D-day에 반영한다.

### 4.2 업무 진행 기록

헤드라인 바로 아래에 최근 진행 기록과 자연어 메모 입력을 둔다.

사용자가 입력한 원문은 그대로 저장한다. AI는 원문을 다음 후보로 구조화한다.

- 일정
- 결정
- 변경
- 후속 작업
- 참고 메모

후보에는 제안 분류, 연결될 날짜 또는 업무 요소와 근거를 표시한다. 사용자가 확인한 후보만 일정, 결정 기록 또는 후속 작업에 반영한다. AI 분석이 실패해도 원문 메모는 유실되지 않으며 사용자가 직접 기록 유형을 선택할 수 있다.

### 4.3 공식 기준과 업무 메모리

진행 기록 아래를 두 영역으로 나눈다.

#### 공식 기준

회사에서 배부하거나 공식 문서 체계로 확인된 자료만 포함한다.

- 사규
- 지침
- 매뉴얼
- 업무 절차서
- 공식 서식

각 항목에는 문서명, 발행 조직, 버전 또는 시행일, 적용 근거를 표시한다.

#### 업무 메모리

동일하거나 유사한 업무에서 축적된 경험 자료를 포함한다.

- 작년·재작년 업무 문서
- 과거 최종 산출물
- 과거 일정과 단계 기록
- 사용자가 확인한 개인 업무 요령
- 이전 담당자가 남긴 확정 메모

각 항목에는 연도, 업무와의 관계, 작성 주체와 출처를 표시한다. 개인 메모는 공식 기준처럼 보이지 않도록 구분한다.

공식 기준과 업무 메모리는 검색 결과가 비슷하더라도 같은 목록에 섞지 않는다.

### 4.4 만들어야 할 결과물

공식 기준과 업무 메모리 아래에 결과물 작성 영역을 둔다.

#### 기존 반복 업무

- 회사 기안 템플릿을 적용한다.
- 과거 승인 문서의 구조를 참고한다.
- 올해 확인된 값만 채운다.
- 확인되지 않은 수치, 일정, 업체명은 빈칸 또는 `확인 필요`로 둔다.
- 작년 문장을 그대로 복사하지 않고 구조와 근거만 재사용한다.

#### 신규 업무

- 회사 템플릿과 공식 기준을 이용해 빈 초안을 시작한다.
- 과거 자료가 없음을 `처음 진행하는 업무`로 명시한다.
- AI는 검토 가능한 초안, 작성 항목과 체크리스트를 제안한다.
- 사용자가 요청하면 공기업 문체와 회사 문서 톤으로 다듬는다.
- AI가 만든 결과를 즉시 결재 가능한 완성본으로 표현하지 않는다.

초안 생성이 실패해도 사용자는 회사 템플릿을 직접 열어 작성할 수 있어야 한다.

### 4.5 완료 및 보관

헤드라인의 `업무 완료`를 누르면 완료 확인 화면을 연다.

완료 전 다음 항목을 확인한다.

- 최종 산출물
- 준공일 또는 개인 완료일
- 적용한 공식 기준
- 확정된 결정과 변경 기록
- 미완료 후속 작업

미완료 후속 작업이 있어도 사용자가 확인하면 완료할 수 있다. 남은 항목은 완료 묶음에 포함해 다음 업무 메모리에서 확인할 수 있게 한다.

완료가 확정되면 다음 변화가 일어난다.

1. 업무의 세부 단계가 `완료`로 바뀐다.
2. 업무가 `내 업무 > 진행`에서 빠진다.
3. 업무가 `내 업무 > 완료`에 남는다.
4. 완료 작업대는 읽기 전용으로 다시 열 수 있다.
5. 최종 산출물, 진행 기록, 적용 기준과 일정 이력을 완료 묶음으로 만든다.
6. 완료 묶음의 문서와 기록을 클라우드에서 조회할 수 있게 한다.
7. 완료 묶음은 다음 연도 또는 다음 담당자가 동일 업무를 시작할 때 업무 메모리 후보가 된다.

완료는 업무 데이터의 삭제나 물리적 이동이 아니라 상태 전환과 완료 스냅숏 생성으로 구현한다.

## 5. 데이터 경계

작업대는 다음 독립 단위를 업무 ID로 연결한다.

### 업무

- 업무 ID
- 작업명
- 세부 단계
- 담당 조직과 담당자
- 계약 여부
- 설계 발송일
- 준공일
- 개인 완료일

### 공식 기준 연결

- 문서 ID
- 문서 유형
- 발행 조직
- 시행일 또는 버전
- 적용 근거
- 확인 상태

### 업무 메모리 연결

- 자료 ID
- 연도
- 원본 업무 ID
- 자료 유형
- 작성 주체
- 연결 이유
- 사용자 확인 상태

### 진행 기록

- 원문
- 작성자와 작성 시각
- AI 제안 분류
- 연결 후보
- 사용자 확인 결과

### 결과물

- 회사 템플릿 ID와 버전
- 초안 버전
- 근거 문서 연결
- 확인 필요 항목
- 최종 산출물

### 완료 묶음

- 완료 시각
- 완료자
- 완료 당시 세부 단계
- 기준 날짜
- 최종 산출물
- 확정 진행 기록
- 적용 공식 기준
- 남은 후속 작업

## 6. 신뢰와 권한

- 사용자는 모든 AI 제안의 출처를 열어볼 수 있어야 한다.
- 공식 기준은 문서 권한이 있는 사용자만 본문을 열 수 있다.
- 문서 접근 권한이 없으면 제목과 존재 여부까지만 표시하고 본문은 노출하지 않는다.
- 개인 업무 요령은 작성자와 공유 범위를 표시한다.
- 일정 후보와 AI 분류는 확정 정보와 시각적으로 구분한다.
- 완료 묶음은 임의로 덮어쓰지 않고 변경 이력을 남긴다.

## 7. 빈 상태와 오류 처리

- 기준일 없음: `일정 미정 · 날짜 추가`
- 공식 기준 없음: `연결된 공식 기준 없음`
- 과거 자료 없음: `처음 진행하는 업무`
- 결과물 없음: 회사 템플릿 선택 또는 빈 초안 시작
- 진행 기록 없음: 첫 메모 입력 안내
- AI 분석 실패: 원문 저장 후 직접 분류 제공
- 초안 생성 실패: 회사 템플릿 직접 편집 제공
- 문서 권한 없음: 권한 안내와 접근 요청 경로 제공
- 완료 자료 부족: 누락 항목을 보여주되 사용자의 최종 완료 판단을 막지 않음

## 8. 모바일 구조

390px에서는 데스크톱의 의미 순서를 그대로 세로로 쌓는다.

1. 헤드라인
2. 업무 진행 기록
3. 공식 기준
4. 업무 메모리
5. 만들어야 할 결과물
6. 완료 관련 정보

공식 기준과 업무 메모리의 구분은 유지하되 두 칸을 한 열로 바꾼다. 문서 전체 가로 스크롤은 만들지 않는다.

## 9. 검증 계약

### 화면과 데이터

- 계약 전 업무는 확정된 설계 발송일 기준 D-day를 표시한다.
- 계약 후 업무는 확인된 준공일 기준 D-day를 표시한다.
- 단계에는 공통 `진행 중` 문구 없이 세부 단계만 표시한다.
- 일정 후보가 사용자 확인 전에 헤드라인 D-day로 승격되지 않는다.
- 공식 기준과 업무 메모리가 서로 다른 영역과 데이터 유형으로 유지된다.
- 개인 업무 요령은 공식 기준으로 표시되지 않는다.

### AI와 편집

- 진행 메모 원문이 AI 분석 성공 여부와 관계없이 저장된다.
- AI 분석 결과는 사용자 확인 전 후보 상태다.
- 기존 업무는 회사 템플릿과 과거 문서 구조를 이용해 초안을 시작한다.
- 신규 업무는 공식 기준과 회사 템플릿에서 초안을 시작한다.
- 확인되지 않은 값은 임의 생성하지 않고 `확인 필요`로 남는다.
- 공기업 문체 교정은 사용자 행동으로 실행된다.

### 완료와 보관

- 완료된 업무는 진행 목록에서 제외된다.
- 완료된 업무는 `내 업무 > 완료`에 남는다.
- 완료 작업대는 읽기 전용으로 열 수 있다.
- 완료 묶음의 문서와 기록을 클라우드에서 조회할 수 있다.
- 남은 후속 작업이 완료 묶음과 다음 업무 메모리에 보존된다.

### 접근성과 반응형

- 세부 단계, D-day, 후보와 확정 상태를 색상만으로 구분하지 않는다.
- 주요 행동은 키보드로 실행할 수 있다.
- 완료 전 확인 화면은 포커스 이동과 복귀가 가능하다.
- 390px에서 정해진 의미 순서를 유지한다.
- 390px에서 문서 전체 가로 넘침이 없다.

## 10. 이번 구현 범위

다음 구현 단계에서는 `product-ui`의 업무 작업대, 내 업무 목록, 클라우드 빈 화면과 관련 브라우저 상태를 이 설계에 맞게 구체화한다.

실제 회사 문서, 계약 시스템, 권한 시스템과 백엔드 저장소는 이후 API 연결 단계에서 같은 데이터 경계를 사용한다. 구현 단계에서는 각 데이터가 없을 때 정직한 빈 상태와 사용자 확인 흐름을 먼저 완성한다.
