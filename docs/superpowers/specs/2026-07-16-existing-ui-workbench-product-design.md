# 기존 UI 기반 업무 작업대 제품 설계

> 승인일: 2026-07-16
> 대상: `demo/app.html`
> 시각 정본: `b8452d1:demo/app.html`의 제품 프로토타입 v3
> 상태: 사용자 승인 완료, 구현 계획 작성 전

## 1. 결정 요약

제품의 중심 단위를 기능이 아니라 **업무 건**으로 바꾼다. 그러나 새 대시보드 디자인을 덮어쓰지 않는다. 사용자가 직접 설계한 기존 `demo/app.html`의 홈, 좁은 중앙 폭, 흰 배경, 얇은 경계선, 파란 포인트, 절제된 카드와 전환 방식을 시각 정본으로 유지한다.

최종 정보 구조는 다음과 같다.

```text
홈 만능 입력
  → 내 업무(목록·달력)
    → 업무 작업대
      → 기안 집중 화면
```

기존 달력·메모·기안은 동등한 기능 탭이 아니다.

- 달력은 업무 건을 찾는 보기 방식이다.
- 메모는 선택한 업무 건의 진행 기록이다.
- 기안은 선택한 업무 건의 최종 산출물을 만드는 행동이다.
- 작업대는 할 일, 근거, 기록, 결과물을 연결하는 제품의 중심 화면이다.

## 2. 왜 이 구조인가

### 검토한 대안

1. **작업대를 네 번째 기능 탭으로 추가**
   기존 변경은 적지만 기능 간 맥락 단절이 그대로 남는다.
2. **홈 아래 한 페이지에 모든 기능 배치**
   전체를 한눈에 볼 수 있지만 정보 밀도가 높고 모바일과 실사용성이 나빠진다.
3. **업무 건 중심으로 기존 기능 재구성**
   기존 UI 정체성을 보존하면서 지시부터 완료까지 같은 업무 맥락을 유지한다.

3번을 채택한다. 사용자가 절감하려는 시간은 기능을 찾는 시간이 아니라, 한 업무를 수행하며 Todo와 과거 자료, 메모와 산출물을 다시 조립하는 시간이다.

## 3. 제품 목표와 범위

### 목표

- 직원이 상급자의 지시 한 줄을 놓치지 않고 업무 건으로 정리한다.
- 반복 업무에서 작년 자료와 이번 할 일을 한 번에 연결한다.
- 현재 해야 할 행동과 그 행동의 근거를 같은 자리에서 확인한다.
- 질문, 메모, 기안 요청이 선택한 업무 맥락을 잃지 않는다.
- 발표에서 `팀장님이 일을 시키면 여기부터 열겠다`는 공감을 만든다.

### 1차 범위

- 반복 업무 70%를 우선한다.
- 대표 시나리오는 `2026년 순환수 펌프 정비공사 추진 보고`다.
- 신규 업무 30%는 빈 업무 건을 만들고 기한·완료 조건을 확인하는 수준까지만 다룬다.
- 데이터와 행동은 시연용 브라우저 상태로 구현한다.
- 선택적인 로컬 `/api/ask` 연결 외에 실제 백엔드가 동작하는 것처럼 표현하지 않는다.

### 제외 범위

- 팀원 저장소 또는 백엔드 변경
- 자동 결재, 자동 제출, 사람을 대신한 최종 판단
- 직원 행동 평가나 감시 기능
- 온톨로지, 인제스트, 벡터 검색 같은 기술 용어의 직원 화면 노출
- 발표용 `demo/index.html`, `demo/jarvis.html`, `demo/admin.html` 수정

## 4. 시각 설계 원칙

기존 GitHub Pages 화면을 기준으로 다음을 유지한다.

- 약 830px 중심의 읽기 폭과 넉넉한 여백
- 흰 배경, 연한 회색 면, 얇은 선
- 파란색 단일 행동 강조
- 작은 캡슐 탭과 절제된 라운드 카드
- 굵은 제목과 낮은 정보 밀도
- 홈의 타자 인트로, 만능 입력, 오프닝 영상
- `시연용 목업 · 샘플 데이터` 고지
- 다크 모드와 모바일 반응형

홈을 둘러싼 네 개의 오브젝트도 제거하지 않고 업무 건 중심으로 역할을 명확히 한다.

| 기존 오브젝트 | 새 역할 | 클릭 결과 |
|---|---|---|
| 왼쪽 위 포스트잇 | 가장 임박한 업무 알림 | 해당 업무 작업대 |
| 오른쪽 위 업무 일기예보 | 도래한 반복 업무 | 내 업무의 반복 업무 보기 |
| 왼쪽 아래 오늘의 메모 | 최근 업무 기록 2건 | 각 기록이 속한 업무 작업대 |
| 오른쪽 아래 근거 문서 연결 | 선택 업무의 대표 근거 | 해당 업무 작업대의 근거 위치 |

홈 아래 예시 질문의 보조 문구는 `달력·메모장·기안 작성` 같은 기능명을 제거하고 `내 업무`, `업무 작업대`, `기안 집중 화면`이라는 실제 착지점을 표시한다. 즉, 홈의 인상과 배치는 유지하되 기능 백화점처럼 보이던 라벨만 업무 흐름에 맞게 교정한다.

홈 오브젝트가 참조할 업무는 다음 규칙으로 결정한다.

- `가장 임박한 업무`는 완료되지 않았고 기한이 있는 업무를 기한 오름차순으로 정한다. 기한이 같으면 명시적 우선순위, 제목, 업무 ID 순으로 안정적으로 정렬한다.
- 기한 미정 업무는 임박 업무 계산에서 제외하고 내 업무 목록에 `기한 확인 필요`로 표시한다.
- 기한이 있는 진행 업무가 없으면 포스트잇은 `기한이 정해진 업무가 없습니다`를 보여주고 내 업무 목록을 연다.
- `선택 업무`는 마지막으로 방문한 유효 업무다. 저장된 값이 없거나 삭제되었으면 가장 임박한 업무를 사용한다.
- 유효한 업무가 하나도 없으면 근거 오브젝트는 `연결된 업무 없음` 상태로 비활성화하고 `내 업무 보기`만 제공한다.

앞서 만든 `f690f68`의 넓은 2열 대시보드, 큰 D-day 박스, 별도 상단 내비게이션은 시각 정본으로 사용하지 않는다. 해당 구현에서는 업무 데이터 모델, 되돌리기, 딥링크, 접근성 검증 아이디어만 재사용할 수 있다.

## 5. 화면 구조

### 5.1 홈 — 말부터 시작

현재 홈 디자인과 검색창을 유지한다. 직원은 질문·지시·메모·기안 모드를 고르지 않고 자연어를 입력한다.

검색창 바로 아래에는 기존의 절제된 링크 문법으로 `내 업무 전체 보기 →`를 둔다. 자연어를 입력하지 않아도 키보드와 포인터로 `#work/list`에 진입할 수 있어야 한다.

홈 입력이 판단하는 것은 두 가지다.

1. 무엇을 하려는가: 질문, 지시, 메모, 할 일, 기안, 업무 전체 보기
2. 어느 업무인가: 기존 업무 건 또는 새 업무

명확하면 바로 해당 업무 맥락으로 이동한다. 불명확하면 의도나 대상 중 모르는 한 가지만 한 번 확인한다.

홈에서 작업대에 바로 들어간 경우 브라우저 뒤로가기는 홈으로 돌아간다. 명확한 기안 요청으로 기안 화면을 바로 연 경우에는 화면에 노출하지 않더라도 `홈 → 해당 업무 작업대 → 기안` 순서의 이력을 만든다. 따라서 첫 뒤로가기는 작업대, 두 번째 뒤로가기는 홈으로 돌아간다.

복합 문장의 의도는 다음 우선순위로 결정한다.

1. 대상 없는 `내 업무`, `이번 주 할 일`은 업무 전체 보기
2. `찾아줘`, `어디`, `알려줘`, 물음표 같은 명시적 질문은 질문
3. `팀장님이`, `올리래`, `하라고 했어` 같은 전달 지시는 지시
4. `확정`, `결정`, `변경`, `합의`는 진행 기록
5. `초안 써줘`, `기안 작성해줘`, `보고서 만들어줘` 같은 생성 동사는 기안
6. `확인해야 함`, `요청하기`, `챙기기`, 기한 표현은 할 일 후보
7. 나머지는 의도 확인

따라서 `펌프 기안 자료 찾아줘`는 기안이 아니라 질문이고, `일정은 5월로 확정`은 Todo가 아니라 결정 기록이다. 이 두 문장은 회귀 테스트 계약에 포함한다.

### 5.2 내 업무 — 일을 찾는 입구

현재 달력 화면의 시각 구조를 유지하고, 업무 건을 찾는 화면으로 역할을 넓힌다.

- `목록 보기 | 달력 보기`를 기존 캡슐 탭 문법으로 제공한다.
- 표시 단위는 개별 Todo가 아니라 업무 건이다.
- 업무 건에는 제목, 지시자, 마감, 현재 단계, 다음 행동, 반복 여부를 보여준다.
- 달력에는 업무 건의 마감 또는 주요 일정만 표시한다.
- 기존 `업무 카드` 행동은 업무 작업대를 연다.
- 기존 `기안 시작`은 작업대를 건너뛰지 않고 해당 업무 작업대의 결과물 영역으로 연결한다.

내 업무의 경로는 `#work/list`와 `#work/calendar`로 구분하며 전환 상태가 새로고침 후에도 유지된다.

### 5.3 업무 작업대 — 일을 수행하는 중심 화면

작업대는 기존 중앙 폭과 카드 문법을 유지한 단일 업무 화면이다. 정보 순서는 다음과 같다.

1. **업무 맥락**
   업무 제목, 원래 지시, 지시자, 기한, 현재 단계, 완료 조건
2. **지금 할 일**
   가장 먼저 처리할 한 행동과 연결된 근거 자료
3. **전체 체크리스트**
   나머지 할 일, 완료 상태, 업무의 진행률
4. **진행 기록**
   메모, 결정, 협업 회신, 변경 이력
5. **만들어야 할 결과물**
   산출물 이름, 필요한 첨부, 기안 시작, 제출 전 주의

Todo와 자료를 별도 탭으로 나누지 않는다. 각 Todo 아래에 `근거 · 작년 서식`, `근거 · 수치 자료`, `근거 · 주의 이력`처럼 직접 연결한다.

작업대 상단에는 기존 작은 검색 입력과 같은 문법의 `이 업무에 이어서 말하기`를 둔다. 여기서는 대상 업무가 고정되어 있으므로 의도만 판단한다.

### 5.4 기안 집중 화면 — 결과물을 완성

기존 기안 화면의 회사 문서 형태를 재사용한다.

- 선택한 업무 제목, 근거 자료, 진행 메모를 그대로 가져온다.
- 작년 양식에서 유지한 값과 올해 확인해야 할 값을 구분한다.
- 임시 저장과 제출 전 점검을 제공한다.
- 뒤로가기는 같은 업무 작업대로 돌아간다.
- 기안은 전역 기능 탭이나 독립 문서함으로 노출하지 않는다.

## 6. 입력 처리와 피드백

### 홈 입력

| 입력 | 결과 |
|---|---|
| `팀장님이 다음 주까지 펌프 정비계획 올리래` | 펌프 반복 업무를 연결하고 작업대를 연다 |
| `작년 펌프 정비 추진 보고 찾아줘` | 펌프 작업대에서 근거 답변과 자료를 보여준다 |
| `운영부 일정은 5월 둘째 주로 확정` | 대상 업무를 한 번 확인한 뒤 진행 기록에 붙인다 |
| `이번 주 내가 해야 할 일` | 내 업무의 목록 또는 달력 보기를 연다 |
| `펌프 추진 보고 기안 작성해줘` | 펌프 업무를 고정하고 기안 집중 화면을 연다 |

### 작업대 입력

- 질문은 연결 자료의 근거 답변으로 표시한다.
- 결정과 변경은 진행 기록에 추가한다.
- 행동 문장은 할 일 후보로 추가한다.
- 명확한 기안 요청은 같은 업무의 기안 집중 화면을 연다.
- 애매한 입력은 질문·메모·할 일 중 한 번만 확인한다.

### 적용 정책

| 입력 유형 | 기본 처리 | 상태 변경 | 취소 방법 |
|---|---|---|---|
| 업무 전체 보기 | 내 업무로 이동 | 없음 | 브라우저 뒤로가기 |
| 질문 | 근거 답변과 자료 표시 | 답변 표시만 변경 | 답변 닫기 또는 뒤로가기 |
| 명확한 지시 | 기존 반복 업무 활성화 또는 새 업무 생성 | 즉시 적용 | 되돌리기 |
| 결정·진행 메모 | 대상이 명확하면 진행 기록에 추가 | 즉시 적용 | 되돌리기·대상 변경 |
| 행동 문장 | `할 일 후보`로 추가 | 후보 상태로 저장, 진행률에는 미반영 | 반영·삭제 |
| 할 일 후보의 `반영` | 정식 체크리스트로 승격 | 진행률과 다음 행동에 반영 | 되돌리기 |
| 기안 요청 | 해당 업무의 기안 집중 화면 열기 | 문서 제출 없음 | 같은 업무 작업대로 복귀 |
| 의도 또는 대상이 애매함 | 모르는 항목만 한 번 확인 | 확인 전 변경 없음 | 입력 다시 쓰기 |

`대상 변경`은 방금 입력으로 생긴 변경만 원래 업무에서 되돌리고 새 대상에 한 번 적용한다. 기존에 그 업무에 있던 Todo, 메모, 지시는 건드리지 않는다.

### 처리 결과

분류명보다 대상과 변경을 보여준다.

```text
“운영부 일정은 5월 둘째 주로 확정”
→ 순환수 펌프 정비공사에 결정 기록을 추가했습니다.
   대상 변경 · 되돌리기
```

새 업무를 만들 때 확인되지 않은 기한은 임의 날짜로 채우지 않고 `기한 미정`으로 표시한다.

## 7. 업무 데이터 모델

목업의 최소 업무 건은 다음을 가진다.

```text
업무 ID
제목과 원래 지시
지시자, 기한, 현재 단계, 완료 조건
Todo 목록과 완료 상태
Todo별 연결 근거 ID
과거 자료와 자료 역할
진행 기록
최종 산출물과 기안 상태
반복 여부
```

진행률과 다음 행동은 정적 숫자가 아니라 Todo 완료 상태에서 계산한다. Todo를 완료하면 다음 미완료 행동이 자동으로 올라오며 되돌리면 이전 상태가 복원된다.

## 8. 상태·탐색·오류 처리

- 경로는 `#home`, `#work/list`, `#work/calendar`, `#workbench/{id}`, `#draft/{id}`를 사용한다.
- 내 업무에서 진입하면 뒤로가기는 `기안 → 같은 업무 작업대 → 이전 내 업무 보기`를 복원한다.
- 홈에서 바로 작업대에 진입하면 뒤로가기는 홈을 복원한다.
- 홈에서 기안으로 직행하면 내부 이력을 `홈 → 작업대 → 기안`으로 구성한다.
- 외부에서 딥링크로 직접 연 경우 브라우저 이력을 조작해 가짜 이전 화면을 만들지 않는다. 대신 화면 안의 `내 업무로`, `같은 업무 작업대로` 버튼으로 복귀한다.
- 존재하지 않는 업무 ID, 초기화로 삭제된 신규 업무 ID, 손상 저장 복구 후 사라진 ID를 열면 전용 `업무를 찾을 수 없습니다` 상태를 보여준다.
- 무효 딥링크를 첫 번째 샘플 업무로 조용히 바꾸지 않으며 요청된 URL도 임의로 정상 업무 ID로 치환하지 않는다.
- 무효 상태에는 `내 업무로`와 `홈으로` 행동을 제공한다. `#draft/{id}`가 무효여도 빈 기안을 만들지 않는다.
- 입력 연결, Todo 변경, 메모 추가, 새 업무 생성은 마지막 행동을 되돌릴 수 있다.
- 대상 변경 시 원래 업무에 적용한 변경을 취소한 뒤 새 대상에 한 번만 적용한다.
- 샘플 원본은 코드 안의 불변 기본값으로 유지한다.
- Todo, 메모, 새 업무, 기안 임시 저장은 버전이 포함된 `localStorage` 키에 저장하여 새로고침과 탭 종료 후에도 유지한다.
- 화면 상단의 `샘플 초기화`는 저장된 변경을 지우고 기본 샘플로 복원한다. 초기화 전에 한 번 확인한다.
- 저장 스키마 버전이 달라지거나 데이터가 손상되면 기본 샘플로 안전하게 복구한다.
- `localStorage`를 사용할 수 없는 환경에서는 현재 탭의 메모리 상태로 동작하고 기능 전체를 막지 않는다.
- 로컬 엔진 연결은 `localhost` 또는 루프백 주소만 허용한다.
- 엔진 URL과 질문 원문은 `localStorage`에 저장하지 않는다.
- 엔진 지연 응답은 더 최근 입력이나 되돌리기를 덮어쓰지 못한다.
- 엔진이 없거나 실패하면 샘플 응답을 유지하고 오류 화면으로 막지 않는다.

## 9. 반응형과 접근성

- 데스크톱은 기존 중앙 폭을 유지하고 필요한 구간만 제한적으로 2열을 사용한다.
- 모바일은 한 열이며 제목, 기한, 지금 할 일이 첫 화면 안에 나타난다.
- 가로 스크롤을 만들지 않는다.
- Todo 버튼은 행동 내용을 포함한 접근성 이름을 가진다.
- Todo의 근거 버튼을 누르면 해당 자료 카드로 포커스가 이동하고 상태 영역이 이를 알린다.
- 모든 주요 화면 전환 후 제목으로 포커스를 옮기되 불필요한 시각 테두리는 만들지 않는다.
- 키보드만으로 입력, 업무 선택, Todo 완료, 근거 확인, 기안 복귀가 가능해야 한다.

## 10. 구현 전환 원칙

1. `b8452d1:demo/app.html`의 시각 구조를 구현 기준으로 복원한다. `origin/main`이 이후 움직여도 이 기준은 바꾸지 않는다.
2. 현재 홈, 오프닝 영상, 샘플 데이터, 다크 모드를 먼저 보존 검증한다.
3. 기능별 라우팅을 업무 건 상태 모델로 교체한다.
4. 현재 달력 화면을 내 업무 화면으로 승격하고 목록 보기를 추가한다.
5. 기존 업무 카드에서 작업대를 열도록 연결한다.
6. 메모 UI를 작업대 진행 기록으로 흡수한다.
7. 기존 기안 UI를 업무별 기안 집중 화면으로 연결한다.
8. 연결 결과, 대상 변경, 되돌리기, 딥링크를 추가한다.
9. 정적 계약과 실제 브라우저 E2E를 통과한 뒤 GitHub에 푸시한다.

## 11. 인수 조건

- GitHub Pages 첫 화면이 기존 홈 UI와 동일한 인상을 유지한다.
- 상단 또는 주요 메뉴에 메모·기안이 독립 제품처럼 노출되지 않는다.
- 내 업무에서 목록과 달력으로 동일한 업무 건을 볼 수 있다.
- 펌프 업무 카드를 누르면 업무 작업대가 열린다.
- 작업대 첫 화면에서 지금 할 일과 그 근거를 함께 확인한다.
- 작업대 입력은 선택한 업무를 잃지 않는다.
- 입력 처리 후 연결된 업무와 변경 내용을 확인하고 대상 변경 또는 되돌리기를 할 수 있다.
- 기안은 선택한 업무에서 열리고 같은 업무로 돌아온다.
- 새 업무의 미확인 기한은 `기한 미정`으로 남는다.
- 새로고침 후에도 Todo, 메모, 기안 임시 저장이 유지되고 `샘플 초기화`로 복원할 수 있다.
- 데스크톱과 390px 모바일에서 가로 오버플로가 없다.
- 발표용 HTML 3개와 팀원 저장소는 변경되지 않는다.
- 정적 테스트, 전체 E2E, `git diff --check`가 통과한다.

- Todo와 근거의 연결 단위
- 협업 대기 상태와 담당자 표시
- 신규 업무가 반복 업무로 전환되는 시점
- 직원이 자동 분류를 거부감 없이 수정하는 표현
