# 유지보수 절차 로컬 시연 fixture 설계

## 목적

비식별 변환본 `sanitized.md`에서 대표 업무인 **정기점검보수 기본계획 수립** 한 건을 구조화해 Product UI의 질문·근거 흐름으로 시연한다. 원문과 변환 결과 파일은 로컬에만 유지하고 Git에는 포함하지 않는다.

## 선택한 접근법

### A. 로컬 전용 fixture 오버레이 — 채택

- 변환 스크립트와 API 라우팅·검증 코드는 추적한다.
- 생성 결과는 `product-ui/fixtures/local-maintenance/`에 저장하고 Git에서 제외한다.
- 기존 `fixture`, `live`, `auto` 계약은 바꾸지 않는다.
- 로컬 오버레이가 없으면 기존 `not-found` fixture로 안전하게 복귀한다.

장점은 실제 변환 데이터를 Git에 싣지 않으면서도 백엔드와 동일한 응답 계약으로 UI를 검증할 수 있다는 점이다. 이후 팀원 백엔드가 같은 문서를 인제스트하면 프론트 코드를 바꾸지 않고 fixture를 재캡처할 수 있다.

### B. 전체 문서 즉시 백엔드 인제스트 — 보류

17,691줄 전체를 한 번에 처리할 수 있지만 `REVIEW / MEDIUM` 판정의 수동 보안검토와 팀원 저장소 변경 승인이 필요하다. 시연 준비의 선행 작업으로는 범위가 크다.

### C. 원문을 프론트 저장소에 포함 — 제외

원문 노출, 저장소 용량, 구조적 재식별 위험 때문에 사용하지 않는다.

## 범위

시연 질문은 다음 의미를 갖는 표현을 지원한다.

- 올해 정기점검보수 기본계획 수립 방법
- 정기점검보수 계획에 포함할 항목
- 계획 수립 시 확인할 과거 기록·협의·승인·안전 항목

응답은 다음 범주의 구조화 정보만 제공한다.

- 적용 범위와 제외 범위
- 계획에 포함할 작업 항목·공정·예산 범주
- 과거 기록·설비 상태·장애 이력·제작사 자료·검사 정보 확인
- 관련 부서 협의와 종합 검토
- 변경·보고·사후 정산·자료관리 체크리스트

기관명, 실제 개정번호·시행일, 마스킹 토큰, 이미지 이름, 법령 세부 조문, 원문 장문은 출력하지 않는다.

## 구성 요소

### 로컬 변환기

`tools/build-local-maintenance-fixture.js`

- 입력 경로를 명시적으로 받아 로컬 Markdown만 읽는다.
- `기본계획 수립` 절의 존재와 필수 하위 항목을 검증한다.
- 원문 전체를 복사하지 않고 정해진 API 응답 구조로 필요한 항목만 만든다.
- 출력 전 금지 패턴과 잔존 마스킹 토큰을 검사한다.
- 네트워크 요청을 하지 않는다.

### 로컬 출력

`product-ui/fixtures/local-maintenance/`

- `manifest.json`: 생성 시각·원본 해시·로컬 전용 표시
- `ask/maintenance-plan.json`: 근거 기반 질문 응답
- `documents/PROC-MAINT-31100.json`: 근거 서랍용 구조화 절차 문서

전체 폴더는 `.gitignore`로 제외한다.

### API 어댑터

`product-ui/api-client.js`

- 유지보수 계획 질문을 로컬 ask fixture로 연결한다.
- 로컬 문서 ID를 로컬 상세 fixture로 연결한다.
- 로컬 파일이 없을 때는 기존 not-found 응답으로 복귀한다.
- 펌프·계약 fixture와 live API 동작은 변경하지 않는다.

## 데이터 흐름

1. 로컬 `sanitized.md`를 변환기에 전달한다.
2. 변환기가 대표 절을 검증하고 로컬 전용 JSON을 생성한다.
3. Product UI를 `?data=fixture`로 실행한다.
4. 사용자가 유지보수 계획 질문을 입력한다.
5. API 어댑터가 로컬 ask fixture를 반환한다.
6. 사용자가 근거를 열면 로컬 구조화 절차 문서를 표시한다.
7. 로컬 fixture가 없는 환경에서는 기존 미검색 응답을 표시한다.

## 보안 경계

- `sanitized.md`, `변환결과_REVIEW.json`, 생성된 local-maintenance fixture, `.env`는 Git에 포함하지 않는다.
- 변환기는 원문 경로를 로그나 생성물에 기록하지 않는다.
- 원본의 SHA-256만 로컬 provenance에 저장한다.
- 생성물은 로컬 시연용이며 팀원 백엔드나 외부 API로 자동 전송하지 않는다.
- 팀원 저장소 반영은 별도 승인과 수동 검토 후 새 브랜치에서 진행한다.

## 오류 처리

- 입력 파일 없음: 생성 중단과 사용법 출력
- 대표 절 없음: 생성 중단
- 필수 구조 누락: 생성 중단
- 금지 패턴 또는 마스킹 토큰 잔존: 생성 중단
- 로컬 fixture 없음: UI에서 기존 not-found fixture 사용
- JSON 계약 불일치: 기존 API 응답 검증에서 실패 처리

## 검증

- 변환기가 원문을 Git 추적 경로에 복사하지 않는지 확인한다.
- 유지보수 질문이 로컬 ask fixture로 라우팅되는지 확인한다.
- 로컬 파일이 없으면 not-found로 복귀하는지 확인한다.
- 근거 문서 ID와 상세 fixture가 연결되는지 확인한다.
- 기존 펌프 질문·계약 데이터·fixture 외부 요청 0건 계약이 유지되는지 확인한다.
- `git status`에서 로컬 생성물과 `.env`가 나타나지 않는지 확인한다.

## 완료 기준

- 로컬 시연에서 유지보수 계획 질문에 근거 답변이 표시된다.
- 근거 서랍에서 구조화된 절차와 체크리스트를 확인할 수 있다.
- 원문과 생성 fixture가 Git 추적 대상에 포함되지 않는다.
- 기존 Product UI 회귀 검사가 통과한다.
