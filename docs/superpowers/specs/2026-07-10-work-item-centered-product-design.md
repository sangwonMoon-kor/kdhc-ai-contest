# 업무 건 중심 제품 설계

## 배경

기존 제품 프로토타입은 검색, 메모, 달력, 기안을 각각 기능으로 보여줬다. 사용자가 실제로 겪는 흐름을 다시 살펴보니 기능보다 먼저 정해야 할 것이 있었다. 직원은 상급자의 지시를 받은 뒤 Todo와 과거 자료를 함께 확인해야 한다.

검색과 메모는 서로 다른 제품 기능이 아니다. 하나의 업무를 시작하고 진행 중인 맥락을 남기는 입력 방식이다. 제품의 기본 단위는 입력창이나 문서가 아니라 `업무 건`으로 확정한다.

## 승인된 제품 정의

> 상급자의 업무 지시를 받는 순간, 해야 할 일과 관련 과거 자료를 한 화면에 정리해 주는 개인 업무 비서.

직원은 업무 프로세스를 처음부터 작성하지 않는다. 시스템이 과거 반복 업무를 기준으로 처리 순서, Todo, 참고 자료, 협조처, 주의 이력을 먼저 준비한다. 직원은 이번에 달라진 부분을 확인하고 고친다.

## 해결할 첫 장면

사용자는 부장이나 팀장에게 업무 지시를 받은 직원이다. 지시는 대개 업무명과 기한만 담고 있다. 직원은 일을 시작하기 전에 다음 내용을 파악한다.

- 무엇을 언제까지 제출해야 하는가
- 작년에는 어떤 문서와 서식을 사용했는가
- 누구와 협의해야 하는가
- 이번에 바꿔야 할 수치와 일정은 무엇인가
- 과거에 반려되거나 지적된 부분은 무엇인가

제품은 이 준비 작업을 하나의 업무 건으로 묶는다.

## 핵심 객체

업무 건은 다음 정보를 가진다.

| 영역 | 내용 |
|---|---|
| 지시 | 요청자, 기한, 산출물, 원문 |
| 반복 맥락 | 업무 유형, 이전 수행 시점, 이전 업무 건 |
| 진행 | 현재 단계, 다음 행동, 전체 체크리스트 |
| 자료 | 과거 결재문서, 서식, 규정, 감사 및 반려 이력 |
| 협업 | 협조 부서, 확인할 사람, 선결 조건 |
| 메모 | 결정, 변경, 추가 지시, 현장 정보 |
| 완료 | 최종 산출물, 증빙, 다음번 주의 사항 |

문서는 업무 건의 근거다. Todo는 업무 건의 진행 상태다. 메모는 진행 중 새로 생긴 맥락이다.

## 1차 범위

PoC는 반복 업무 70%를 대상으로 한다. 최근 수행 이력이 있어 과거 흐름을 재사용할 수 있는 업무를 우선한다.

신규 사업 30%는 같은 업무 건 구조를 사용하지만 이번 설계 범위에서는 `처음 진행하는 업무`로만 구분한다. 조사, 기획, 대안 비교 같은 신규 사업 지원은 후속 설계로 미룬다.

## 사용자 흐름

1. 직원이 상급자의 지시를 만능 입력창에 적는다.
2. 시스템이 반복 업무와 기존 업무 건을 찾는다.
3. 새 업무 건에 기한, 산출물, Todo 후보, 과거 자료를 채운다.
4. 직원이 업무 건을 열어 다음 행동과 관련 자료를 함께 확인한다.
5. 진행 중 질문과 메모는 현재 업무 건에 붙는다.
6. 초안, 사전 점검, 결재 상신을 거쳐 업무를 완료한다.
7. 최종 산출물과 메모는 다음 반복 주기의 기준이 된다.

## 한 페이지 정보 구조

화면은 세 영역으로 나눈다.

### 입력과 업무 목록

- 상단 만능 입력
- 진행 중인 업무 건
- 기한과 현재 상태
- 반복 업무 도래 알림

### 선택한 업무 건

- 지시 요약과 완료 조건
- 현재 단계
- 지금 할 일
- 전체 체크리스트

### 연결된 맥락

- 과거 자료
- 협조처와 선결 조건
- 반려 및 감사 주의 사항
- 진행 메모와 최종 산출물

Todo와 자료는 별도 탭으로 멀리 나누지 않는다. 사용자가 다음 행동을 보는 자리에서 해당 행동에 필요한 근거를 함께 확인할 수 있어야 한다.

## 입력 처리 원칙

| 입력 유형 | 제품 반응 |
|---|---|
| 질문 | 현재 또는 과거 업무 건에서 근거와 답을 찾는다 |
| 지시 | 새 업무 건과 Todo 후보를 만든다 |
| 메모 | 현재 업무 건에 진행 맥락을 추가한다 |
| 애매한 입력 | 검색할지 업무로 남길지 한 번만 확인한다 |

사용자에게 질문, 지시, 메모 모드를 먼저 고르게 하지 않는다. 분류 결과는 되돌릴 수 있어야 하며, 확신이 없을 때만 확인한다.

## 직원 수용 원칙

- 조직 지식 축적보다 당장 내 일을 끝내는 효용을 먼저 보여준다.
- 직원에게 별도 인수인계 문서나 업무 분류표를 작성하게 하지 않는다.
- 자동 분류와 과거 자료 연결은 직원이 확인하고 고칠 수 있어야 한다.
- 개인 평가와 감시로 이어질 수 있는 표현과 화면을 배제한다.
- 시스템이 모르는 내용은 과거 자료가 없다고 밝힌다.

## 경진대회 전달 구조

발표는 흔한 장면에서 시작한다.

> 팀장님이 다음 주까지 계획을 올리라고 했다. 무엇부터 해야 하지?

입력 한 번 뒤에 Todo와 작년 자료가 함께 나타나는 순간을 대표 장면으로 삼는다. 업무 승계와 조직 지식은 이 경험이 반복된 뒤 생기는 결과로 설명한다.

## 이번 설계에서 제외한 범위

- 신규 사업의 조사와 기획 자동화
- 자율 에이전트와 자동 결재
- 직원이 직접 만드는 복잡한 프로세스 편집기
- 팀원 백엔드 저장소 수정
- 직원 행동을 평가하는 관리자 분석

## 수용 기준

- 지시 한 줄에서 업무 건 한 개가 만들어진다.
- 업무 건에서 Todo와 관련 과거 자료를 동시에 확인할 수 있다.
- 반복 업무는 이전 업무 건의 흐름과 자료를 가져온다.
- 질문과 메모는 선택한 업무 건에 연결된다.
- 사용자가 자동 분류를 고치거나 되돌릴 수 있다.
- 초기 화면에 온톨로지나 데이터 자산 같은 기술 용어를 노출하지 않는다.
- 신규 업무는 과거 자료가 없음을 숨기지 않는다.

## 후속 순서

이 설계가 문서 검토를 통과하면 다음 순서로 진행한다.

1. 업무 건 상세 화면 와이어프레임
2. 반복 업무 카드의 필수 정보 확정
3. 만능 입력의 지시, 질문, 메모 피드백 설계
4. 기존 `demo/app.html`을 업무 건 중심으로 재배치하는 구현 계획
5. 팀 회의에서 API 연결과 저장 및 확인 단계 합의
