# 아이디어 회의: 만능 입력에서 업무 건 중심으로

> 작성일: 2026-07-10
> 정리: 문상원과 대화하며 정리
> 상태: **제품 관점 확정, UI 설계 전**
> 관련: [demo/app.html](../demo/app.html), 팀원 엔진 `creationy/jikmu-memory`

## 1. 이번에 바뀐 생각

처음에는 직원이 궁금한 것을 묻는 검색창에서 시작했다. 여기에 메모까지 받을 수 있으면 검색창과 메모장을 하나로 합칠 수 있다고 봤다.

그런데 검색과 메모는 제품의 중심이 아니다. 둘 다 업무를 시작하거나 진행 중인 내용을 남기는 방법이다. 직원이 실제로 정리해야 하는 대상은 문서나 메모가 아니라 **한 건의 업무가 지시에서 완료까지 흘러가는 과정**이다.

이 제품의 중심 문장은 다음과 같다.

> 상급자의 업무 지시를 받는 순간, 해야 할 일과 관련 과거 자료를 한 화면에 정리해 주는 개인 업무 비서.

캐치프레이즈는 나중에 바꿀 수 있다. 지금 확정하는 것은 제품이 어느 순간에 누구를 돕는지다.

## 2. 직원이 실제로 겪는 시작점

공기업 직원은 부장이나 팀장에게 업무 지시를 받으면 곧바로 일을 시작하기 어렵다. 먼저 기한과 산출물을 확인하고, 작년 자료나 유사 사례를 찾고, 협조할 부서와 해야 할 일을 다시 정리한다.

보통 다음 순서로 움직인다.

1. 지시를 받는다.
2. 무엇을 언제까지 내야 하는지 파악한다.
3. 작년 문서와 기존 서식을 찾는다.
4. 이번에 바뀐 값과 협의할 부서를 확인한다.
5. 할 일을 순서대로 정리한다.
6. 초안을 만들고 검토와 결재를 거친다.
7. 최종 문서와 진행 중 알게 된 내용을 남긴다.

현재는 Todo와 자료 찾기가 따로 떨어져 있다. 직무 메모리는 이 둘을 하나의 업무 건 안에서 연결한다.

## 3. 제품의 기본 단위: 업무 건

업무 건은 문서 한 개나 할 일 한 줄이 아니다. 지시, 과거 자료, Todo, 협의 내용, 진행 메모, 최종 산출물을 한데 묶은 작업 단위다.

```text
지시
→ 업무 파악
→ 과거 자료 확인
→ Todo와 협조처 정리
→ 작성과 실행
→ 검토와 결재
→ 완료 자료와 메모 축적
```

각 요소의 역할은 다음처럼 정리한다.

| 요소 | 업무 건 안에서 하는 역할 |
|---|---|
| 지시 | 요청자, 기한, 산출물을 정한다 |
| Todo | 이 업무를 끝내기 위한 다음 행동을 보여준다 |
| 과거 자료 | 작년에는 어떤 순서와 서식으로 처리했는지 알려준다 |
| 메모 | 진행 중에 생긴 결정, 변경, 주의 사항을 붙인다 |
| 협조처 | 누구와 무엇을 확인해야 하는지 보여준다 |
| 산출물 | 결재문서와 증빙을 완료 결과로 남긴다 |

문서는 업무 건에 붙는 근거이고, Todo는 업무 건의 진행 상태다. 검색과 메모는 업무 건을 만들거나 보충하는 입력 방식이다.

## 4. 1차 범위: 반복 업무 70%

1차 PoC는 반복 업무에 집중한다. 실제 업무를 반복 70%, 신규 사업 30%로 본다면 먼저 70%를 해결한다.

반복 업무는 작년 업무 건을 올해 업무 건으로 가져올 수 있다. 기존 처리 순서, 사용한 서식, 협조 부서, 반려 이력, 완료 자료가 이미 있기 때문이다. 직원은 처음부터 프로세스를 작성하지 않는다. 시스템이 과거 흐름을 먼저 준비하고 직원은 올해 달라진 부분만 확인한다.

신규 사업은 이번 범위에서 깊게 다루지 않는다. 과거 자료가 없는 업무는 `처음 진행하는 업무`로 표시하고 빈 업무 건으로 시작한다. 신규 업무의 조사, 기획, 의사결정 지원은 후속 단계에서 다룬다.

## 5. 첫 사용 시나리오

직원이 만능 입력창에 다음과 같이 적는다.

> 팀장님이 다음 주까지 순환수 펌프 정비공사 계획을 올리라고 했어.

시스템은 입력을 검색어나 메모로만 처리하지 않고 하나의 업무 건으로 정리한다.

### 이번 업무

- 요청자: 팀장
- 기한: 다음 주
- 완료 조건: 추진계획 결재 상신
- 유형: 매년 반복되는 업무

### 지금 할 일

- 작년 추진계획 확인
- 운영부와 정지 가능 기간 협의
- 올해 진동값과 견적 갱신
- 추진계획 초안 작성
- 반려 및 감사 이력 사전 점검
- 결재 상신

### 함께 볼 자료

- 2025년 추진계획
- 작년 결재문서와 첨부 서식
- 산출근거 누락 반려 이력
- 구두 설계변경 감사 지적
- 전임자가 남긴 진행 메모

### 바로 할 수 있는 행동

- 할 일에 추가
- 작년 자료 열기
- 작년 양식으로 시작

이 화면에서 직원은 자료를 다시 찾고 Todo를 별도로 옮길 필요가 없다. 이번 업무를 끝내는 데 필요한 것만 한곳에서 확인한다.

## 6. 한 페이지 UI의 기준

화면은 기능별 메뉴보다 현재 진행 중인 업무를 중심으로 구성한다.

1. 상단에는 지시, 질문, 메모를 모두 받는 입력창을 둔다.
2. 화면 한쪽에는 진행 중인 업무 건과 기한을 보여준다.
3. 선택한 업무 건에는 현재 단계와 다음 할 일을 가장 먼저 보여준다.
4. Todo 옆에는 그 행동과 연결된 과거 자료와 주의 사항을 붙인다.
5. 진행 중 남긴 메모와 최종 산출물은 같은 업무 건에 쌓인다.

초기 화면에서 온톨로지, 조직 지식, 데이터 자산 같은 말은 앞세우지 않는다. 직원이 먼저 확인해야 하는 것은 `지금 무엇을 해야 하는지`, `작년 자료가 어디 있는지`, `이번에 무엇을 바꿔야 하는지`다.

## 7. 직원이 참여하는 이유

직원에게 인수인계 자료를 따로 만들거나 조직 지식을 입력해 달라고 요구하지 않는다. 지시받은 일을 처리하려고 서비스를 쓰면 그 과정이 다음번에 다시 쓸 수 있는 업무 건으로 남는다.

직원이 얻는 이익은 바로 보인다.

- 지시받은 일을 놓치지 않는다.
- 작년 자료를 찾는 시간을 줄인다.
- Todo와 참고 자료를 따로 관리하지 않는다.
- 과거 반려와 감사 지적을 제출 전에 확인한다.
- 다음 반복 주기에는 올해 업무 건에서 다시 시작한다.

업무 승계와 조직 지식은 직원에게 추가로 요구하는 일이 아니라 평소 업무 처리의 결과다. 경진대회에서도 `회사를 위해 데이터를 쌓아 달라`가 아니라 `내가 받은 일을 더 쉽게 끝낸다`라는 경험을 먼저 보여준다.

## 8. 만능 입력의 새 역할

만능 입력은 여전히 필요하지만, 제품 자체를 뜻하지는 않는다. 업무 건으로 들어가는 하나의 입구다.

| 입력 | 처리 결과 |
|---|---|
| 질문 | 현재 업무 건이나 과거 업무에서 답과 근거를 찾는다 |
| 지시 | 새 업무 건을 만들고 기한, 산출물, Todo 후보를 정리한다 |
| 메모 | 진행 중인 업무 건에 결정, 변경, 할 일을 추가한다 |
| 애매한 입력 | 검색할지 업무로 남길지 한 번만 확인한다 |

직원은 질문인지 메모인지 먼저 고르지 않는다. 내용을 적으면 시스템이 적절한 업무 건에 연결하고, 확신이 없을 때만 짧게 되묻는다.

## 9. 기술 구현과 팀 회의

팀원 엔진에는 질문, 힌트 구조화, 문서 인제스트 경로가 이미 있다. 이 문서에서는 API 연결 방식이나 저장 정책을 결정하지 않는다. 기술 구현은 다음 주 팀 회의에서 논의한다.

회의에서 확인할 내용은 다음과 같다.

- 지시 문장에서 기한과 산출물, 반복 업무를 어디까지 찾을 수 있는가
- 기존 자료를 하나의 업무 건으로 묶어 돌려줄 수 있는가
- 진행 중 메모를 어느 업무 건에 연결할 것인가
- 자동 분류 결과를 언제 직원에게 확인받을 것인가

기술 구조보다 먼저 합의할 것은 `업무 건이 제품의 기본 단위`라는 점이다.

## 10. 이번 단계에서 하지 않는 것

- 신규 사업 30%를 위한 조사와 기획 지원
- 직원에게 업무 프로세스를 처음부터 입력시키는 화면
- 문서함과 Todo를 별도 제품처럼 운영하는 구조
- 자동 결재나 사람을 대신한 최종 판단
- 직원 평가나 감시로 오해받을 수 있는 개인 행동 분석
